# Логика генерации постов (Этап 5)

Документ описывает, как на **Этапе 5** выбранная тема (`Topic`) превращается в черновик публикации (`Post`). Реальный AI и сеть **не** используются: тексты собираются по детерминированным шаблонам форматов и помощникам текста. Автопостинг и согласование через Telegram на этом этапе не выполняются.

## Как `Topic` превращается в `Post`

Точка входа — `PostGenerationService` (`backend/app/services/post_generation_service.py`). Для одной темы шаги такие:

1. **Найти тему и проект.** По `topic_id` берётся `Topic`; если темы нет — `TopicNotFoundError` (API → `404`). По `topic.project_id` берётся `Project`.
2. **Определить формат.** Приоритет: формат из запроса (`recommended_format`) → рекомендованные форматы кластера из `topic_taxonomy` → эвристика `infer_format_from_topic` по ключевым словам заголовка. По умолчанию — `product`.
3. **Подобрать медиа.** `PostMediaSelectionService.select_media_for_topic` ищет подходящий `MediaAsset` (см. ниже).
4. **Собрать CTA и тексты.** `build_cta(project_slug, format_name)` и `generate_platform_texts` дают `telegram_text`, `vk_text`, `instagram_text`.
5. **Добавить SEO и хэштеги.** `seo_keywords` берутся из `Topic.seo_keywords`; `hashtags` строит `build_hashtags`.
6. **Определить статус.** `draft`, если медиа подобрано; `needs_media`, если нет.
7. **Сохранить `Post`** в таблицу `posts` и вернуть `PostGenerationResult` (`post`, `selected_media_asset_id`, `needs_media`, `warnings`, `generation_notes`).

Пакетный режим `generate_weekly_posts` сначала вызывает выбор тем (`TopicSelectionService`), берёт верхние `weeks × posts_per_week` рекомендованных тем и генерирует пост для каждой.

## Форматы постов

Таксономия форматов — `backend/app/services/post_template_taxonomy.py`. Для каждого формата заданы `purpose`, `structure`, `tone`, `cta_type`.

| Формат | Назначение | Структура | Тип CTA |
|--------|-----------|-----------|---------|
| `expert` | Показать экспертизу, помочь разобраться | hook → explanation → practical_value → soft_cta | soft |
| `product` | Представить изделие и применение | product_intro → use_cases → benefits → cta | direct |
| `technology` | Объяснить технологию нанесения | what_it_is → when_to_use → advantages → cta | direct |
| `case` | Показать решение задачи на практике | task → solution → result → cta | direct |
| `faq` | Ответить на частые вопросы | question → answer → recommendation → cta | soft |
| `selling` | Прямое предложение услуги | offer → benefits → proof → direct_cta | direct |

`infer_format_from_topic(title, cluster, recommended_formats=None)` выбирает формат: сначала первый известный из `recommended_formats`, затем по ключевым словам (например, «шелкография»/«dtf» → `technology`, «как …» → `expert`, «ошибки»/«когда»/«… или …» → `faq`), иначе — `product`.

## Версии текста под Telegram, VK, Instagram

Метод `generate_platform_texts(project_slug, topic, format_name, cta)` собирает три версии из общих «блоков тела» формата:

- **Telegram** — самый развёрнутый: все абзацы тела + CTA.
- **VK** — структурный и продающий: вводный абзац + список «чем поможем» + CTA.
- **Instagram** — короткий и визуальный: сокращённый «крючок» (`shorten_text`) + CTA + строка хэштегов.

Тексты различаются по структуре и длине под площадку. Формулировки сдержанные: без обещаний «лучшие», «№1», «гарантируем».

### Пример (TEEON, тема «Футболки с логотипом на заказ», формат `product`)

Telegram-версия примерно такая:

```
Футболки с логотипом на заказ — один из самых востребованных вариантов в категории «футболки».

Такие изделия заказывают для мероприятий, команд, промоакций, welcome-наборов и подарков клиентам. До запуска стоит определиться с материалом, тиражом и способом нанесения логотипа.

В TEEON можно собрать решение под задачу компании — от небольшого пробного тиража до крупной партии.

Напишите нам — подберём изделие, нанесение и рассчитаем тираж.
```

## CTA и хэштеги

Помощники — `backend/app/services/post_text_helpers.py`.

- **CTA.** `build_cta(project_slug, format_name)` хранит по 3 варианта для TEEON и «Фабрики сувениров»; вариант выбирается детерминированно по формату (`product`/`selling`/`case` — более прямые, `expert`/`faq`/`technology` — мягче). Для неизвестного проекта — нейтральные CTA без брендовых деталей.
- **Хэштеги.** `build_hashtags(project_slug, topic_title, cluster, seo_keywords)` строит тематические хэштеги из SEO-ключей, кластера и заголовка, затем добавляет брендовые: `#teeon` для TEEON, `#фабрикасувениров` для «Фабрики сувениров». `clean_hashtag` убирает пробелы и лишние символы (кириллица разрешена, `ё→е`, нижний регистр), отбрасывает дубли и пустые.

Пример: `["#футболкислоготипом", "#футболкиназаказ", "#корпоративныефутболки", "#футболки", "#teeon", "#корпоративныймерч"]`.

## Подбор медиа и `needs_media`

`PostMediaSelectionService.select_media_for_topic(db, topic)` (`backend/app/services/post_media_selection_service.py`):

- рассматриваются только медиа проекта со статусом `approved` или `approved_video`;
- сопоставление по тегам групп `products`/`technologies`/`details`/`categories`/`use_cases`/`topics`: совпадение с кластером темы (важнее) и со словами заголовка (учитывается единственное/множественное число);
- внешние стоки (`source_type == external_stock`) допускаются только при подтверждённой лицензии (не `needs_review`);
- **не** используются: `new` (если есть `approved`), `rejected`, `needs_reshoot`, `needs_license_review`;
- если подходящего медиа нет — возвращается `(None, warnings)`, и пост получает статус **`needs_media`** (его нужно дополнить фото/видео — см. рекомендации по досъёмке Этапа 3).

## Статусы поста

Статусы и переходы — `backend/app/services/post_status_service.py`: `draft`, `needs_review`, `approved`, `scheduled`, `published`, `rejected`, `needs_media`. Полная таблица переходов — в [`Докс/06_Модель_данных.md`](./06_Модель_данных.md) (раздел «Post после Этапа 5»). Неизвестный статус → `422`, запрещённый переход → `409`.

## API и команды

REST API `/posts` (`backend/app/api/posts.py`):

- `GET /posts?project_id=&topic_id=&status=&limit=&offset=` — список;
- `POST /posts/generate/topic/{topic_id}` — сгенерировать пост по теме (тело `PostGenerationRequest` опционально); `404`, если темы нет;
- `POST /posts/generate/weekly-plan` — сгенерировать посты на неделю(и) (`WeeklyPostGenerationRequest`); `404`, если проекта нет;
- `GET /posts/{post_id}` — получить пост (`404`);
- `PATCH /posts/{post_id}` — ручная правка (`title`, тексты, `hashtags`, `seo_keywords`, `media_asset_id`);
- `PATCH /posts/{post_id}/status` — смена статуса (`404`/`422`/`409`).

CLI:

```bash
make generate-post topic_id=1
make generate-weekly-posts project_slug=teeon
```

## Ограничения Этапа 5

- реальный AI **не** подключён — тексты шаблонные и детерминированные;
- автопостинг **не** выполняется (соцсети не подключены);
- согласование через Telegram **не** выполняется (это Этап 6);
- внешние сетевые запросы и обращения к Яндекс Диску при генерации не делаются;
- расписание публикаций на этом этапе не задаётся.
