# Архитектура

Документ описывает архитектуру системы **AI-SMM-бот** — сервиса автоматического ведения соцсетей для проектов TEEON и «Фабрика сувениров». Здесь приведена обзорная схема потоков данных и описание каждого компонента: от FastAPI-приложения и базы данных до AI-модуля, Telegram-интерфейса, автопостинга, планировщика и аналитики.

## Обзорная схема потоков данных

```
                          ┌────────────────────────────────────────────┐
                          │              ВНЕШНИЕ ИСТОЧНИКИ                │
                          │  Яндекс Диск (медиа) · AI-провайдер (LLM)    │
                          │  VK API · Telegram API · Instagram API        │
                          └───────────────┬──────────────────────────────┘
                                          │  (через app/integrations)
                                          ▼
   ┌───────────────┐        ┌─────────────────────────────────┐        ┌───────────────┐
   │   Человек     │◀──────▶│        BACKEND (FastAPI)         │◀──────▶│  PostgreSQL   │
   │ (SMM-команда) │  TG     │  app.main → api → services →    │ ORM     │  (SQLAlchemy  │
   │               │ управл. │  models/schemas · ai · scheduler │ Alembic │   + миграции) │
   └───────────────┘ согл.   └───────────────┬─────────────────┘        └───────────────┘
                                             │
                                             ▼
                                       ┌───────────┐
                                       │   Redis    │  кэш · очереди задач · блокировки
                                       └───────────┘

  ОСНОВНОЙ ПОТОК (по этапам жизненного цикла поста):

  [Яндекс Диск] ──скан──▶ MediaTaggingService ──▶ MediaAsset (теги, статус)
                                                        │
  направления человека ──▶ TopicSelectionService ──────┤ (учёт наличия медиа,
        + SEO/тренды/сезонность/конкуренты/статистика   │  SEO, истории постов)
                                                        ▼
                                                    Topic (priority_score)
                                                        │
                                              PostGenerationService (AI)
                                                        ▼
                          Post (telegram_text · vk_text · instagram_text · hashtags)
                                                        │
                                          Telegram-согласование (человек)
                                                ┌───────┴────────┐
                                          одобрено            отклонено/правки
                                                │                  │
                                          Scheduler ◀──────────────┘ (возврат на доработку)
                                                │
                                   Автопостинг (VK/TG → Instagram)
                                                │
                                                ▼
                                   AnalyticsService (сбор метрик)
                                                │
                                                └──▶ обратная связь в TopicSelectionService
```

Цикл замкнут: статистика опубликованных постов влияет на дальнейший выбор тем, поэтому система постепенно «учится» на собственных результатах.

## 1. Backend (FastAPI)

Ядро системы — FastAPI-приложение в пакете `backend/app`. Архитектура слоистая, зависимости направлены сверху вниз (api → services → models/integrations).

| Слой | Назначение |
|------|-----------|
| `app.main` | Точка входа: создание `FastAPI`, подключение роутеров, lifespan-хуки (старт/стоп планировщика, прогрев кэша). Здесь же `GET /health → {"status":"ok","service":"ai-smm-bot"}`. |
| `app/api` | HTTP-эндпоинты (роутеры), валидация запросов через схемы, преобразование результатов сервисов в ответы. Тонкий слой без бизнес-логики. |
| `app/services` | Бизнес-логика: `TopicSelectionService`, `MediaTaggingService`, `PostGenerationService`, `AnalyticsService`. Оркеструют работу моделей, интеграций и AI. |
| `app/models` | SQLAlchemy-модели: `Project`, `MediaAsset`, `Topic`, `Post`. Отражают таблицы БД и связи (FK). |
| `app/schemas` | Pydantic-схемы для входа/выхода API и внутренней передачи данных (отделены от ORM-моделей). |
| `app/integrations` | Клиенты внешних сервисов (Яндекс Диск, Telegram, VK, Instagram, AI). Изолируют сетевые вызовы. |
| `app/core` | Конфигурация (Pydantic Settings, чтение `.env`), общие зависимости, логирование. |
| `app/db` | Подключение к БД, фабрика сессий, базовый класс моделей. |
| `app/ai`, `app/scheduler`, `app/utils` | AI-логика и промпты, фоновые задачи, вспомогательные функции (например, разбор имён файлов). |

Принцип: api-слой не обращается к БД и внешним API напрямую — только через сервисы. Сервисы получают сессию БД и клиентов интеграций как зависимости, что упрощает тестирование (моки клиентов) и соответствует правилу «все внешние интеграции — через отдельные клиенты».

## 2. База данных (PostgreSQL + Redis)

**PostgreSQL** — основное хранилище состояния. Доступ через **SQLAlchemy** (ORM-модели в `app/models`), схема версионируется миграциями **Alembic**. Ключевые сущности и связи:

- `Project` — корневая сущность; к ней привязаны медиа, темы и посты.
- `MediaAsset` → `project_id`; хранит путь на Яндекс Диске, теги (JSONB), статус и время последнего использования.
- `Topic` → `project_id`; тема с `priority_score`, `business_priority`, `seo_keywords` (JSONB), кластером и статусом.
- `Post` → `project_id`, `topic_id`, `media_asset_id`; тексты под каждую соцсеть, хэштеги, расписание и статус.

JSONB-поля (`tags`, `seo_keywords`, `hashtags`) позволяют хранить гибкие наборы данных без отдельных таблиц на раннем этапе.

**Redis** дополняет PostgreSQL и используется для трёх задач:

1. **Кэш** — результаты сканирования Яндекс Диска, ответы AI, агрегированная статистика (снижение нагрузки и расходов на API).
2. **Очереди задач** — постановка фоновых заданий (сканирование, генерация, публикация), которые исполняет планировщик/воркеры.
3. **Блокировки** — распределённые локи, чтобы один и тот же пост не был опубликован дважды и параллельные задачи не конфликтовали.

## 3. Яндекс Диск как источник медиа

Команда загружает фото и видео в папки проектов на Яндекс Диске. Бот через клиент в `app/integrations` периодически **сканирует** Диск: читает имена файлов и структуру папок, по которой определяет проект. Для каждого нового или изменённого файла создаётся/обновляется запись `MediaAsset` с `yandex_disk_path`.

Имя файла несёт первичную семантику и разбирается утилитой (рабочая логика уже реализована). Пример: «Худи с карманом с шелкографией и жаккардами.jpg» → проект **TEEON**; изделие — худи; технология — шелкография; детали — карман, жаккард; темы — «худи с логотипом», «корпоративный мерч», «шелкография на худи». Эти данные ложатся в поля `title`, `description`, `tags`. Поле `last_used_at` помогает избегать повторного использования одних и тех же изображений подряд.

## 4. AI-модуль

AI-логика сосредоточена в `app/ai` и отвечает за три задачи:

- **Анализ изображений** — уточнение тегов и описаний `MediaAsset` сверх того, что извлечено из имени файла (распознавание изделия, технологии, деталей).
- **Генерация тем** — на основе направлений человека, SEO-запросов, трендов, сезонности, активности конкурентов и статистики прошлых постов формируются конкретные `Topic` с оценкой приоритета.
- **Генерация текстов** — для каждой соцсети создаётся свой вариант (`telegram_text`, `vk_text`, `instagram_text`) с учётом тона площадки, плюс `hashtags` и `seo_keywords`.

Конкретный AI-провайдер **абстрагирован за клиентом** в `app/integrations` — смена модели или поставщика не затрагивает бизнес-логику. Все **промпты вынесены в `app/ai/prompts.py`**, что позволяет править формулировки без изменения кода вызовов. Токены провайдера хранятся только в `.env`.

## 5. Telegram-интерфейс управления и согласования

Telegram — основной канал взаимодействия человека с ботом. Через него команда:

- задаёт **стратегические направления** (что продвигаем в текущем периоде);
- получает на **согласование** сгенерированные посты с предпросмотром текстов и прикреплённым медиа;
- **одобряет, отклоняет или отправляет на доработку** (через inline-кнопки);
- запрашивает статус, статистику и ручной запуск задач.

Решение человека меняет `status` соответствующего `Post`: одобренный пост передаётся планировщику на публикацию, отклонённый — возвращается в генерацию с учётом правок. Клиент Telegram изолирован в `app/integrations`; этот же клиент используется и для автопостинга в Telegram-канал.

## 6. Автопостинг (VK / Telegram → Instagram)

После одобрения и наступления времени из `scheduled_at` пост публикуется в соцсети. На первом этапе поддерживаются **VK** и **Telegram**, далее добавляется **Instagram**. Для каждой площадки — свой клиент в `app/integrations` с единым внутренним интерфейсом публикации (текст + медиа).

Особенности:
- В пост идёт текст из соответствующего поля (`vk_text`, `telegram_text`, `instagram_text`) и медиа по `media_asset_id`.
- Перед публикацией берётся Redis-блокировка, чтобы исключить дубли.
- После успешной публикации проставляется `published_at` и обновляется `status`; ошибки логируются и задача может быть повторена.
- **Запрещено** публиковать внешние картинки как собственные кейсы — это контролируется полями `source_type`/`license_type` у `MediaAsset`.

## 7. Планировщик (scheduler)

Модуль `app/scheduler` запускает периодические и отложенные фоновые задачи, координируя их через Redis-очереди и блокировки:

- сканирование Яндекс Диска и обновление `MediaAsset`;
- запуск подбора тем и генерации постов по расписанию;
- публикация одобренных постов в назначенное `scheduled_at` время;
- регулярный сбор статистики для аналитики.

Планировщик стартует и останавливается через lifespan-хуки `app.main`. Длительные операции не блокируют HTTP-обработчики API.

## 8. Аналитика и обратная связь

`AnalyticsService` собирает статистику опубликованных постов (охваты, реакции, переходы и т. п.) через клиенты соцсетей и сохраняет агрегаты в БД/кэш. Эти данные используются двояко:

1. **Отчётность** — человек видит, какие темы, форматы и изделия работают лучше.
2. **Обратная связь в выбор тем** — статистика передаётся в `TopicSelectionService` и влияет на `priority_score` будущих тем. Так система усиливает успешные направления и снижает приоритет неэффективных.

## Как компоненты взаимодействуют (сводно)

1. **Скан медиа.** Scheduler → клиент Яндекс Диска → `MediaTaggingService` → `MediaAsset` (с тегами).
2. **Выбор тем.** Направления человека + SEO/тренды/сезонность/конкуренты + наличие медиа + статистика → `TopicSelectionService` (AI) → `Topic`.
3. **Генерация.** `PostGenerationService` (AI, промпты из `app/ai/prompts.py`) → `Post` с текстами под каждую соцсеть и подобранным медиа.
4. **Согласование.** Telegram-клиент показывает пост человеку → решение меняет `status`.
5. **Публикация.** Scheduler по `scheduled_at` → клиенты VK/Telegram/Instagram (под Redis-локом) → `published_at`.
6. **Аналитика.** `AnalyticsService` собирает метрики → обновляет приоритеты тем → цикл повторяется.

Все внешние вызовы идут только через `app/integrations`, конфигурация и секреты — через Pydantic Settings и `.env`, состояние — в PostgreSQL, а кэш, очереди и блокировки — в Redis.
