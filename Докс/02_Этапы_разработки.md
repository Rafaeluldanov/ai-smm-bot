# Этапы разработки

Документ описывает поэтапный план разработки **AI-SMM-бота** — системы автоматического ведения соцсетей проектов компании (TEEON и Фабрика сувениров). Каждый этап опирается на результаты предыдущего и завершается набором проверок.

## Главное правило перехода между этапами

> **Переход к следующему этапу возможен только когда `make check` проходит успешно.**

`make check` запускает три проверки подряд:

| Проверка | Инструмент | Что контролирует |
|----------|-----------|------------------|
| Линт + формат | `ruff check` | Стиль кода, неиспользуемые импорты, форматирование |
| Типы | `mypy` | Корректность аннотаций типов |
| Тесты | `pytest` | Логика, регрессии, контракты API |

Дополнительные правила, действующие на всех этапах: токены и секреты — только в `.env` (никогда в коде); все внешние интеграции — через отдельные клиенты в `app/integrations`; нельзя публиковать внешние картинки как собственные кейсы; сложные действия логируются.

## Этап 1 — Проекты

> **Статус: ✅ Реализовано (2026-06-18)**

- **Цель:** управлять списком проектов компании.
- **Что сделано:**
  - **Alembic.** В корне `alembic.ini`; пакет `backend/alembic/` (`env.py` берёт URL БД из настроек приложения и `target_metadata` из `app.models`; `script.py.mako`; каталог `versions/`). Первая миграция `0001_initial` создаёт таблицы `projects`, `media_assets`, `topics`, `posts`. JSON-поля на PostgreSQL = `JSONB`; внешние ключи с `ON DELETE CASCADE` (`project_id`) и `ON DELETE SET NULL` (`topic_id`, `media_asset_id`); уникальный индекс `ix_projects_slug`, индексы по FK; `created_at`/`updated_at` с `server_default now()`.
  - **DB-зависимость.** `backend/app/api/deps.py` → `get_db()` выдаёт SQLAlchemy `Session` и закрывает её после запроса (делегирует в `app/db/session.py`).
  - **Репозиторий проектов.** `backend/app/repositories/project_repository.py`: `get_project_by_id`, `get_project_by_slug`, `list_projects(active_only=True)`, `create_project`, `update_project` (частичное обновление через `exclude_unset`), `deactivate_project` (ставит `is_active=False`, запись не удаляется). Дубликат slug → `SlugAlreadyExistsError` (API превращает в 409).
  - **Схемы.** `backend/app/schemas/project.py`: `ProjectCreate`, `ProjectUpdate` (все поля опциональны), `ProjectRead` (она же для списка). Валидация slug — нормализация к нижнему регистру: разрешены латиница, цифры, `-` и `_`; значение приводится к нижнему регистру (`TEEON` → `teeon`); пробелы и кириллица отклоняются (422); минимальная длина — 2 символа.
  - **REST API.** `backend/app/api/projects.py` (роутер с префиксом `/projects`, подключён в `main.py`): `GET /projects?active_only=true` (список); `POST /projects` (201; дубль slug → 409; невалидный slug → 422); `GET /projects/slug/{slug}` (404); `GET /projects/{project_id}` (404); `PATCH /projects/{project_id}` (частичное обновление; 404; занятый slug → 409); `POST /projects/{project_id}/deactivate` (404). Все ответы типизированы через `response_model` (`ProjectRead`). Удаления из БД на этом этапе нет.
  - **Сидинг.** `backend/app/scripts/seed_projects.py` + команда `make seed-projects`; идемпотентно (без дублей, печатает отчёт создано/уже существует). Проекты: **TEEON** (slug `teeon`, `website_url=https://teeon.ru`) и **Фабрика сувениров** (slug `fabric-souvenirs`, `website_url=null`).
  - **Makefile.** Добавлены команды `db-up` (`docker compose up -d db redis`), `db-down`, `migrate` (`alembic upgrade head`), `revision` (`make revision message="..."`), `seed-projects`.
  - **Тесты.** Используют SQLite в памяти, поэтому `make check` не зависит от запущенного Docker/PostgreSQL: `conftest.py` создаёт схему и переопределяет зависимость `get_db`. Покрыто: `/health`; Project API (создание, дубль slug → 409, get by id, get by slug, список, фильтр `active_only`, частичное обновление, обновление slug на занятый → 409, деактивация, 404 для несуществующих, 422 для невалидного slug); валидация slug; репозиторий (`get_project_by_slug`, `deactivate` не удаляет запись). Всего 34 теста — все проходят; тесты Этапа 0 (включая `MediaTaggingService`) сохранены.
- **Проверки:** `make format` и `make check` (ruff + mypy на 46 файлах + pytest, 34 теста) проходят успешно. Миграция дополнительно проверена offline-генерацией SQL (корректный PostgreSQL DDL: JSONB, FK, индексы); seed и API проверены вживую.
- **Порядок локального запуска:** 1) `make install`; 2) `cp .env.example .env`; 3) `make db-up`; 4) `make migrate`; 5) `make seed-projects`; 6) `make run`.
- **Критерий готовности:** миграции применяются; после сидинга оба проекта присутствуют в БД; CRUD покрыт тестами; `make check` проходит. ✅

## Этап 2 — Яндекс Диск

> **Статус: ✅ Реализовано (2026-06-18)**

- **Цель:** подключить хранилище медиа и переносить файлы в БД как `MediaAsset`.
- **Что сделано:**
  - **Конфигурация.** В `app/config.py` и `.env.example` добавлены `YANDEX_DISK_TOKEN` (`str | None`, OAuth-токен), `YANDEX_DISK_BASE_URL=https://cloud-api.yandex.net/v1/disk`, `YANDEX_DISK_ROOT_PATH=/SMM_BOT`. Без токена обычный API работает; sync-эндпоинты возвращают `503`.
  - **Клиент Яндекс Диска (read-only).** `backend/app/integrations/yandex_disk/client.py`: REST API Яндекс Диска через `httpx` + OAuth-токен. Ошибки `YandexDiskError` / `YandexDiskAuthError` / `YandexDiskNotFoundError`; dataclass `YandexDiskResource` (`name`, `path`, `type`, `mime_type`, `size`, `modified`, `public_url`; свойства `is_file`/`is_dir`). Методы: `list_resources` (`GET /resources`, парсит `_embedded.items`), `list_files_recursive` (рекурсивно, возвращает только файлы, `max_depth=3`), `get_download_url` (`GET /resources/download` → `href`). Пустой токен → `YandexDiskAuthError`; HTTP 404 → `YandexDiskNotFoundError`; 401/403 → `YandexDiskAuthError`. Файлы целиком на этом этапе не скачиваются; в тестах используется `httpx.MockTransport` (реальный API не дёргается).
  - **Структура хранилища.** На Яндекс Диске: `/SMM_BOT/<папка_проекта>/{01_Входящие_на_разбор, 02_Одобренные_фото, 03_Видео, 04_Внешние_картинки_из_интернета, 05_Использовано_в_постах, 06_Нужно_переснять}`. На Этапе 2 сканируются первые 4 папки.
  - **Mapping папок проектов.** `backend/app/services/project_media_paths.py`: `get_project_disk_root(slug)` (teeon → `/SMM_BOT/01_TEEON`, fabric-souvenirs → `/SMM_BOT/02_Фабрика_сувениров`), `get_default_scan_folders(slug)` → 4 полных пути (`01_Входящие_на_разбор`, `02_Одобренные_фото`, `03_Видео`, `04_Внешние_картинки_из_интернета`). Неизвестный slug → `UnknownProjectError`.
  - **Репозиторий `MediaAsset`.** `backend/app/repositories/media_asset_repository.py`: `get_media_asset_by_id`, `get_media_asset_by_path`, `list_media_assets(project_id, status, limit, offset)`, `create_media_asset`, `update_media_asset`, `upsert_media_asset_from_disk_resource` (бизнес-уникальность по `yandex_disk_path`; возвращает `(asset, action)`, где `action ∈ created/updated/unchanged` — повторная синхронизация не плодит дубликаты).
  - **Модель и миграция.** `MediaAsset` обновлена: `yandex_disk_path` (unique + index); `source_type ∈ {internal, external_stock}` (индекс); `license_type ∈ {company_owned, external_needs_review, commercial_use_required}`; `status ∈ {new, approved, approved_video, needs_license_review, used}`; `tags` заполняются `MediaTaggingService`. Миграция `backend/alembic/versions/0002_media_asset_disk_path_indexes.py`: unique index `ix_media_assets_yandex_disk_path`; индексы `ix_media_assets_status`, `ix_media_assets_source_type`. Совместима с PostgreSQL; миграция `0001` не изменялась; тесты строят схему через SQLite `metadata.create_all`.
  - **Схемы.** `backend/app/schemas/media_asset.py`: `MediaAssetRead`, `MediaAssetCreate`, `MediaAssetUpdate` (все поля опциональны), `MediaAssetSyncResult` (`project_id`, `project_slug`, `scanned_folders`, `found_files`, `created`, `updated`, `skipped`, `errors`).
  - **Сервис синхронизации.** `backend/app/services/yandex_disk_media_sync_service.py`: `YandexDiskMediaSyncService(client, tagging_service)` с методами `sync_project_media(db, project_id)` и `sync_project_media_by_slug(db, slug)`. Логика: найти проект (нет → `ProjectNotFoundError` → 404); получить папки сканирования; для каждой `list_files_recursive` (папка не найдена → запись в `errors` и продолжаем; `YandexDiskAuthError` → прерывание всей синхронизации → 503). Классификация по пути: `04_Внешние_картинки_из_интернета` → `source_type=external_stock`, `license_type=external_needs_review`, `status=needs_license_review`; `02_Одобренные_фото` → `status=approved`; `03_Видео` → `status=approved_video`; иначе (`01_Входящие` и пр.) → `source_type=internal`, `license_type=company_owned`, `status=new`. Теги через `MediaTaggingService.extract_tags_from_file_name(имя)`; upsert по `yandex_disk_path`. Возвращает `MediaAssetSyncResult`.
  - **REST API.** `backend/app/api/media_assets.py` (префикс `/media-assets`, подключён в `main.py`): `GET /media-assets?project_id=&status=&limit=&offset=` → список `MediaAssetRead` (работает без токена); `GET /media-assets/{id}` → 404 если нет; `POST /media-assets/sync/project/{project_id}` → `MediaAssetSyncResult` (404 нет проекта; 503 нет/отклонён токен; 400 неизвестная папка проекта); `POST /media-assets/sync/slug/{slug}` → то же. Реальные запросы к Диску — только при sync. Зависимости `get_yandex_disk_client` / `get_media_sync_service` в `app/api/deps.py` (в тестах подменяются).
  - **Скрипт и команда.** `backend/app/scripts/sync_media.py` + команда `make sync-media project_slug=teeon` (читает `--project-slug`, печатает отчёт `scanned_folders`/`found`/`created`/`updated`/`skipped`/`errors`; без токена — понятное сообщение). Не входит в `make check`.
  - **Тесты.** На SQLite в памяти + mock, без сети/Docker/PostgreSQL: клиент Яндекс Диска (парсинг `_embedded.items`, рекурсия только по файлам, 404 → NotFound, пустой токен → AuthError, download href), `project_media_paths`, `media_asset_repository` (create/get_by_path/upsert без дублей/фильтры), сервис синхронизации (создание `MediaAsset`, идемпотентность, теги для «Худи с карманом с шелкографией и жаккардами.jpg» → изделие: худи / технология: шелкография / детали ⊇ {карман, жаккард}, external → `external_stock` + `needs_license_review`, отсутствующая папка → `errors` но синхронизация продолжается, неизвестный проект → `ProjectNotFoundError`), API media-assets (list/get/404/фильтр/sync с mock/sync 404/sync без токена → 503). Старые тесты Этапов 0 и 1 сохранены. Всего 60 тестов — все зелёные.
- **Проверки:** `make format` и `make check` (ruff + mypy на 51 файле + pytest, 60 тестов) проходят успешно. Миграция `0002` проверена offline-генерацией SQL (корректный PostgreSQL DDL). Live-смоук на временной БД: `GET /media-assets` работает без токена, sync без токена → 503, отсутствующий проект → 404. Docker и реальный Яндекс Диск в среде недоступны — не использовались.
- **Порядок ручной синхронизации (локально):** 1) `make db-up`; 2) `make migrate`; 3) `make seed-projects`; 4) `make sync-media project_slug=teeon`. Для реальной синхронизации нужен `YANDEX_DISK_TOKEN` в `.env`.
- **Критерий готовности:** сканирование папки наполняет таблицу `media_assets` без дублей; повторный запуск не создаёт копий; интеграция замокана в тестах; `make check` проходит. ✅

## Этап 3 — Анализ медиа

> **Статус: ✅ Реализовано (2026-06-18)**

- **Цель:** автоматически описывать и тегировать загруженные медиа.
- **Что сделано:**
  - **Модуль словарей терминов.** `backend/app/services/media_taxonomy.py`: словари `PRODUCT_TERMS` (изделия одежды), `TECHNOLOGY_TERMS` (методы нанесения), `DETAIL_TERMS` (детали — жаккард, бирка, шеврон и пр. отнесены сюда), `MATERIAL_TERMS`, `COLOR_TERMS`, `USE_CASE_TERMS`, `AUDIENCE_TERMS`, `PRODUCTION_WORDS` и правила категорий. Функции: `normalize_text` (нижний регистр, `ё→е`, удаление расширения файла, замена `_ - .` на пробелы), `extract_keywords_by_taxonomy(text)`, `build_topics_from_tags(tags, project_slug)`, `build_seo_keywords_from_tags(tags, project_slug)`, `calculate_tag_confidence(tags)`. Сопоставление морфологическое: точное совпадение токена, по префиксу («жаккардами» → «жаккард»), по общему префиксу с расхождением в хвосте ≤2 символов («шелкографией» → «шелкография»); многословные термины — как подстрока; короткие коды (`dtf`, `dtg`, `uv`, `hr`, `b2b`) — только точное совпадение.
  - **Расширенный `MediaTaggingService`.** `backend/app/services/media_tagging_service.py`: метод `analyze_file_name(file_name, project_slug=None, yandex_disk_path=None, source_type=None)` возвращает богатую структуру; `extract_tags_from_file_name` сохранён для совместимости с Этапом 2; добавлены `suggest_content_topics` и `suggest_seo_keywords`. Структура `tags`: `products`, `technologies`, `details`, `materials`, `colors`, `categories`, `use_cases`, `audiences`, `topics`, `seo_keywords`, `matched_terms`, `confidence` (0.0–1.0), `needs_review` (bool), `review_reasons` (list). Флаг `needs_review=true`, если: `confidence<0.35`; или нет ни `products`, ни `technologies`; или `source_type==external_stock`; или путь содержит `04_Внешние_картинки_из_интернета`. Категории: `apparel` (изделие одежды), `souvenirs` (сувенирный продукт), `branding` (есть технология), `production` (слова цех/производство/станок/процесс), `packaging` (упаковка/коробка), `event_merch` (мероприятие/выставка/конференция/event), `external_reference` (`source_type=external_stock` или путь внешних картинок).
  - **Статусы проверки и безопасные переходы.** `backend/app/services/media_status_service.py`: допустимые статусы `new`, `approved`, `approved_video`, `needs_license_review`, `rejected`, `needs_reshoot`, `used`. Функции `get_allowed_statuses`, `can_transition`, `validate_transition`, `update_media_status`; исключения `InvalidMediaStatusError` (неизвестный статус) и `InvalidMediaStatusTransitionError` (запрещённый переход). Правила переходов: `new → {approved, approved_video, needs_license_review, rejected, needs_reshoot}`; `approved → {used, rejected, needs_reshoot}`; `approved_video → {used, rejected, needs_reshoot}`; `needs_license_review → {approved, rejected}`; `needs_reshoot → {approved, rejected}`; `used → {approved, approved_video}`; `rejected → {approved, needs_reshoot}`.
  - **`MediaAnalysisService`.** `backend/app/services/media_analysis_service.py` (зависимости — `MediaTaggingService`, `MediaStatusService`, репозитории): методы `analyze_media_asset(db, id, save=True)`, `retag_media_asset(db, id)`, `retag_project_media(db, project_id)`, `retag_project_media_by_slug(db, slug)`, `get_tags_summary(db, project_id=None)`, `suggest_shooting_tasks(db, project_id)`. Сводка считает частоты по `products`/`technologies`/`details`/`materials`/`colors`/`categories`/`use_cases`/`audiences`. Рекомендации по досъёмке: для **TEEON** целевые темы (футболка, худи, шоппер, шелкография, dtf, вышивка, жаккард), для **Фабрики сувениров** (кружка, ручка, пакет, уф-печать, тампопечать, гравировка, шелкография); если по теме одобренных медиа меньше 2 — предлагается задача с папкой `06_Нужно_переснять` и списком кадров.
  - **Папки и классификация.** `project_media_paths.get_default_scan_folders` теперь возвращает 5 папок — добавлена `06_Нужно_переснять`; `05_Использовано_в_постах` не сканируется.
