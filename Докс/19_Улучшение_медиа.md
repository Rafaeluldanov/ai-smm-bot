# Улучшение медиа (Media Enhancement)

Документ описывает этап **«Улучшение медиа»** — безопасную локальную подготовку
изображений к публикации. Бот создаёт **улучшенные копии**, не трогая оригиналы,
и **без подключения реального AI**.

## Главный принцип: оригиналы неприкосновенны

- Улучшение всегда создаёт **новую копию** — отдельный файл и запись
  `MediaAssetVariant`. Исходный `MediaAsset` **не изменяется**: ни путь
  (`yandex_disk_path`), ни статус, ни сам файл.
- Файлы на Яндекс Диске **не перезаписываются**. Производные копии хранятся
  локально в `MEDIA_ENHANCEMENT_STORAGE_DIR` (по умолчанию
  `backend/data/enhanced_media`, добавлено в `.gitignore`).

## Что бот делает локально (через Pillow)

- авто-контраст (`auto_contrast`);
- мягкая коррекция яркости и насыщенности (`brightness`);
- баланс белого с **жёстким** ограничением коэффициентов (`white_balance`);
- лёгкий denoise (`denoise`);
- повышение резкости (`sharpen`);
- ресайз (уменьшение до `max_side`, апскейла нет) (`resize`);
- конвертация HEIC/JPEG/PNG/WebP в рабочий формат (`convert`, по умолчанию JPEG).

HEIC поддерживается через `pillow-heif`. Если изображение нельзя открыть (битый
файл/неизвестный формат), процессор возвращает **понятную ошибку**, а не падает.

## Профили обработки

| Профиль | Операции | Назначение |
|---------|----------|------------|
| `social_safe` (по умолчанию) | convert, resize, auto_contrast, brightness, sharpen (мягко) | Безопасная подготовка под соцсети без искажения цвета |
| `product_clean` | то же + `white_balance` + `denoise` + сильнее sharpen | Чище для карточек товара; правки **спорные** → review |
| `minimal` | только convert + resize | Просто привести к рабочему формату/размеру |

## Чего бот НЕ делает без AI

- Не выполняет «умную» ретушь: **удаление грязи/пятен, выравнивание цвета ткани,
  улучшение текстуры, апскейл** — это только **интерфейс-заглушка**
  (`app/ai/image_editing.py`, `StubImageEditor`), реальный AI **не подключён**.
- Локальные фильтры **не имитируют** точечную ретушь и не «прячут» дефекты товара.

**Почему так:** автоматическое скрытие дефектов изделия может ввести покупателя в
заблуждение. Поэтому будущая AI-ретушь обязательно проходит **ручной review**, а не
публикуется автоматически.

## Review для спорных правок

Если применены операции, меняющие реальный цвет/текстуру изделия (`white_balance`,
`denoise`), процессор добавляет предупреждение:

> «Автоулучшение не должно искажать реальный цвет/качество изделия — проверьте
> копию вручную перед публикацией»

и копия получает статус **`needs_review`**. Безопасные правки (`social_safe`)
дают статус `created`. Видео **пропускаются** (только изображения).

## Источник изображения

Байты берутся через `MediaDownloadService`:

- `public://yandex/<slug>/<path>` — публичная папка Яндекс Диска: ссылка на
  скачивание берётся через публичный API (без токена), затем скачиваются байты;
- `external://…` — внешние стоки **не скачиваются** (unsupported);
- приватный путь Яндекс Диска — на этом этапе не поддержан загрузчиком.

В тестах сеть и AI **не вызываются**: публичный клиент и HTTP-транспорт
подменяются (`httpx.MockTransport` / fake-загрузчик).

### Особенность публичного Яндекс Диска

Важная асимметрия между синхронизацией и улучшением:

- **Sync (`sync-public-media`)** использует **только метаданные** из листинга
  публичной папки — файлы не скачиваются.
- **Enhancement** требует **скачать байты**, а для этого нужна прямая ссылка
  (`download_url`).

Для существующих файлов download-эндпоинт обычно отвечает **200**, но для
«неудобных» путей (пробелы, кириллица, отдельные HEIC) Яндекс может вернуть
**404** или **400 (DiskPathFormatError)**. Поэтому
`YandexDiskPublicClient.get_public_download_url` устойчив и работает так:

1. пробует `/public/resources/download` (с ведущим `/` и без него);
2. при **любой** ошибке эндпоинта (404 **и** 400/прочие) или отсутствии `href` —
   находит ресурс листингом родительской папки (`find_public_resource_by_path`,
   листинг идёт **постранично** — в реальных папках бывают сотни файлов) и берёт
   его **прямую** ссылку `download_url` (поле `file` из метаданных листинга);
3. если и так не вышло — понятная ошибка с **каноническим** путём актива:
   «Не удалось получить ссылку на скачивание: …».

Поле `YandexDiskPublicResource.download_url` заполняется из поля API `file`.

> **Важно (устаревшие пути).** Если улучшение пишет «Публичный ресурс не найден»,
> скорее всего файл на диске **переименован/удалён** после синхронизации
> (например, `IMG_5113.HEIC` переименован тегами): путь в БД стал **устаревшим**,
> и скачать такой файл нельзя ничем — это не ошибка кода, а рассинхронизация данных.
> **Сначала сделайте `make sync-public-media project_slug=<slug>`** — он обновит
> `MediaAsset` под текущее состояние папки, после чего улучшайте свежий id.
> Сам sync ещё и покажет такие записи **warning-ом** `Stale public media not found
> on disk: N` (записи при этом не удаляются автоматически — см.
> [`18_Публичная_папка_Яндекс_Диска.md`](./18_Публичная_папка_Яндекс_Диска.md)).

Таким образом downloader для публичных ресурсов сначала пробует download-эндпоинт,
а затем **fallback на `file`/`download_url` из метаданных** — это и чинит
скачивание HEIC для улучшения. Поле `YandexDiskPublicResource.download_url`
заполняется из поля API `file`. Оригиналы при этом не меняются — скачанные байты
идут только в производную копию.

## API

Базовый префикс — `/media-enhancements`.

| Метод и путь | Назначение | Коды |
|--------------|------------|------|
| `GET /media-enhancements` | Список вариантов (фильтры `media_asset_id`/`project_id`/`status`/`variant_type`) | 200 |
| `GET /media-enhancements/summary?project_id=` | Сводка по статусам и типам | 200 |
| `POST /media-enhancements/media/{media_asset_id}/enhance` | Улучшить медиа (создать копию) | 200; 404; 409; 400; 503 |
| `POST /media-enhancements/project` | Пакетно улучшить медиа проекта | 200; 404 |
| `PATCH /media-enhancements/{variant_id}/status` | Сменить статус варианта | 200; 404; 422 |
| `GET /media-enhancements/{variant_id}` | Получить вариант по id | 200; 404 |

Коды: `404` — медиа/вариант/проект не найден; `409` — вариант уже есть и `force=false`;
`400` — формат/источник не поддержан; `503` — загрузчик не настроен; `422` —
неизвестный статус в `PATCH`.

```bash
# Один актив (social_safe). 409 — если уже улучшено (нужен "force": true)
curl -X POST http://localhost:8000/media-enhancements/media/1/enhance \
  -H "Content-Type: application/json" -d '{"profile": "social_safe"}'

# Пакетно по проекту (по умолчанию approved)
curl -X POST http://localhost:8000/media-enhancements/project \
  -H "Content-Type: application/json" \
  -d '{"project_slug": "teeon", "status": "approved", "profile": "social_safe"}'

# Сводка и смена статуса
curl "http://localhost:8000/media-enhancements/summary?project_id=1"
curl -X PATCH http://localhost:8000/media-enhancements/1/status \
  -H "Content-Type: application/json" -d '{"status": "approved"}'
```

## CLI

```bash
make enhance-media media_asset_id=1                  # одно медиа
make enhance-project-media project_slug=teeon        # все approved-медиа проекта
make media-enhancement-summary project_slug=teeon    # сводка по копиям
```

`enhance-project-media` поддерживает `--status all` (улучшить все статусы),
`--profile`, `--force` (пересоздать), `--limit`.

## Настройки (`.env`)

| Переменная | По умолчанию | Назначение |
|------------|--------------|------------|
| `MEDIA_ENHANCEMENT_ENABLED` | `false` | Флаг готовности (показывается в `/health/readiness`) |
| `MEDIA_ENHANCEMENT_STORAGE_DIR` | `backend/data/enhanced_media` | Папка для копий |
| `MEDIA_ENHANCEMENT_MAX_IMAGE_MB` | `25` | Лимит размера входного изображения |
| `MEDIA_ENHANCEMENT_DEFAULT_PROFILE` | `social_safe` | Профиль по умолчанию |
| `MEDIA_ENHANCEMENT_OUTPUT_FORMAT` | `jpg` | Рабочий формат вывода |
| `MEDIA_ENHANCEMENT_JPEG_QUALITY` | `92` | Качество JPEG |

## Модель данных

`MediaAssetVariant` (таблица `media_asset_variants`, миграция
`0010_media_asset_variants`) — см. `Докс/06_Модель_данных.md`.

## Ручная проверка на текущих данных

```bash
make sync-public-media project_slug=teeon       # подтянуть медиа (Тион)
make media-summary project_slug=teeon
# одобрить нужные MediaAsset (PATCH .../status approved) или взять уже approved
make enhance-project-media project_slug=teeon   # создать улучшенные копии
make media-enhancement-summary project_slug=teeon
```

После этого оригинальные `MediaAsset` остаются без изменений, а в
`media_asset_variants` появляются производные копии.
