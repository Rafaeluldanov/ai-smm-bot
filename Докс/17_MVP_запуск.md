# MVP-запуск (Production hardening)

Документ описывает подготовку бэкенда AI-SMM-бота к боевому запуску. Все этапы разработки (0–10) завершены; здесь — что добавлено для готовности к продакшену и что нужно сделать **вручную** перед реальным запуском.

Этот документ — не новый этап разработки и **не** добавляет бизнес-функции. Реальные токены не коммитятся, реальные публикации в тестах не выполняются.

## Что добавлено для готовности (в коде)

- **Readiness-проба** `GET /health/readiness` — сообщает окружение, тип БД и настроенные интеграции, без сети и обращений к БД. В production добавляет предупреждения о незаданных токенах и о SQLite вместо PostgreSQL.
- **Config helper properties** (`app/config.py`): `is_production`, `is_local`, `database_is_sqlite`, `telegram_configured`, `vk_configured`, `yandex_disk_configured`, `ai_configured`.
- **Смоук-проверка** `make smoke` (`app/scripts/smoke_check.py`) — поднимает приложение in-process, проверяет `/health` и `/health/readiness`, печатает сводку окружения без значений секретов.
- **`.gitignore`** — игнорирует `.env`, `.env.*`, `*.secret`, `*.pem`, `*.key`, локальные БД и логи; `.env.example` остаётся в репозитории.
- **`.env.example`** — шаблон со всеми настройками; реальные значения только в `.env` (вне git).

## Эндпоинты проверки

```bash
# Liveness — процесс жив
curl http://localhost:8000/health
# {"status":"ok","service":"ai-smm-bot"}

# Readiness — готовность окружения/интеграций
curl http://localhost:8000/health/readiness
```

Поле `warnings` пустое в `local`; в `production` перечисляет, что не настроено для боевого режима (но сервис всё равно отвечает — он деградирует мягко: эндпоинты, требующие токенов, возвращают понятную ошибку).

## Порядок боевого запуска (вручную)

1. **Секреты.** Заполнить `.env` (или менеджер секретов) реальными значениями и выставить окружение:
   ```dotenv
   APP_ENV=production
   DATABASE_URL=postgresql+psycopg2://<user>:<pass>@<host>:5432/<db>
   YANDEX_DISK_TOKEN=<oauth-токен>
   TELEGRAM_BOT_TOKEN=<токен-бота>
   TELEGRAM_DEFAULT_CHANNEL_ID=@<канал>
   VK_ACCESS_TOKEN=<токен>
   VK_DEFAULT_GROUP_ID=-<id-группы>
   ```
   Реальные токены **никогда** не коммитятся (см. `.gitignore`).
2. **Миграции.** Применить схему на боевой БД: `make migrate` (`alembic upgrade head`, текущий head — `0009_autonomous_runs`).
3. **Сидинг.** `make seed-projects` (идемпотентно создаёт TEEON и «Фабрику сувениров»).
4. **Смоук.** `make smoke` — убедиться, что приложение поднимается и readiness отвечает.
5. **Живой режим публикаций.** Включить реальную отправку в `TelegramPublishingClient`/`VKPublishingClient` (сейчас живой режим отключён и вместо отправки бросается понятная `PublishError`). Это намеренная защита: на MVP включается осознанно.
6. **Запуск автономного режима — постепенно.** Начать с безопасных режимов:
   - `make autonomous-dry-run project_slug=teeon` — посмотреть план без изменений;
   - `make autonomous-run project_slug=teeon` (`semi_auto`) — посты уходят на согласование человеку;
   - только после ручной проверки качества — включать `auto_schedule`/`auto_publish` с явными флагами.

## Что ещё нужно вне бэкенда

- **Мониторинг и алерты:** ошибки публикации, упавшие автономные прогоны, посты `needs_media`, рост `failed`-публикаций.
- **Логирование:** структурированные логи сложных действий (публикация, прогоны), агрегация и хранение.
- **Секреты:** хранение и ротация токенов (vault/secret manager), а не plain `.env` в проде.
- **Резервное копирование:** регулярный backup PostgreSQL и проверка восстановления.
- **UI/админка:** интерфейс для согласования постов (поверх `/post-reviews`) и просмотра прогонов/аналитики.
- **Ручное QA:** прогон на реальном проекте, проверка текстов, медиа и лицензий внешних картинок (внешнее — не наш кейс).

## Критерий готовности MVP

- секреты заданы вне репозитория; `APP_ENV=production`;
- миграции применены на боевой БД; `make smoke` зелёный;
- живой режим публикаций включён осознанно; автономный режим запускается от `dry_run`/`semi_auto`;
- настроены мониторинг, логирование и бэкапы;
- есть админ-интерфейс для ревью; проведено ручное QA;
- `make check` проходит.
