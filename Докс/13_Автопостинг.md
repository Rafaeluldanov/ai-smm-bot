# Автопостинг (Этап 7)

Документ описывает backend-логику **планирования и публикации** одобренных постов в Telegram и VK. Публикация идёт через изолированные клиенты, которые легко подменяются на фейковые в тестах. Реальные сетевые вызовы и Instagram на этом этапе **не** подключены; аналитика эффективности — Этап 8.

## Жизненный цикл: approved → scheduled → published

```
approved ──schedule──► scheduled ──publish (все платформы ok)──► published
                          │                                         ▲
                          │ publish (часть ошибок)                  │
                          └──────────── остаётся scheduled ─────────┘
```

1. **schedule** (`PostPublicationService.schedule_post`): по `approved`/`scheduled` посту создаёт/обновляет `PostPublication` для каждой платформы; `Post.status → scheduled` (и `Post.scheduled_at`, если задан). Публикации получают статус `scheduled` (с датой) или `pending` (без даты — публиковать сразу).
2. **publish** (`publish_post`): публикует пост по платформам. При успехе **всех** платформ без ошибок `Post.status → published` (через `scheduled`), `Post.published_at` проставляется. Если есть ошибки — пост остаётся `scheduled`.
3. **publish-due** (`publish_due_publications`): находит «созревшие» публикации и публикует их (используется планировщиком).

Из каких статусов можно: планировать — `approved`/`scheduled`; публиковать — `approved`/`scheduled`/`published` (последнее — чтобы корректно отдавать `skipped`/`force`). Иначе — `PostNotPublishableError` (API → 409).

## Telegram и VK

Публикация абстрагирована от платформы:

- `PublishRequest` (платформа, `target_id`, текст, медиа, хэштеги, payload) и `PublishResponse` (`external_post_id`, `external_url`, `raw`) — `backend/app/integrations/publishing.py`.
- Клиенты реализуют протокол `PublishingClient.publish_post(request) → PublishResponse`.
- `TelegramPublishingClient` и `VKPublishingClient` — безопасные: реальная отправка включается **только** флагом `live_enabled` (см. ниже), без токена/таргета — `PublishError`. В тестах используется `FakePublishingClient` либо реальный клиент с подменённым HTTP-транспортом (`httpx.MockTransport`).
- `PublicationPlatformRegistry` сопоставляет платформу с клиентом (`get_client`, `get_available_platforms`); неизвестная платформа → `UnknownPlatformError`.

Для Telegram берётся `post.telegram_text`, для VK — `post.vk_text`; хэштеги кладутся в `payload`; `target_id` — из публикации (заданный при планировании) или из настроек по умолчанию (`TELEGRAM_DEFAULT_CHANNEL_ID` / `VK_DEFAULT_GROUP_ID`).

**Медиа.** Если у `MediaAsset` поста есть **одобренный** (`approved`) улучшенный вариант (`MediaAssetVariant`), в запрос кладётся путь к улучшенной копии как `preferred_media_path` (`media_path`), а `payload.media_source = "enhanced_variant"`. Иначе — метаданные оригинала (`payload.media_source = "original"`, `attachment` с `file_name`/`yandex_disk_path`). В `payload` также кладётся `media_kind` (`image` | `video` | `none`).

**Загрузка фото во VK (v0.1.12).** При живой публикации `VKPublishingClient` прикрепляет изображение: читает байты из локального файла улучшенной копии (`media_path`) либо скачивает оригинал из публичной папки Яндекс Диска (`public://yandex/...`), затем `photos.getWallUploadServer` → multipart-upload → `photos.saveWallPhoto` → `attachments=photo{owner_id}_{id}` в `wall.post`.

- **Видео** (`.mov`/`.mp4`/…) пока не загружается: пост уходит текстом, предупреждение — в `PublishResponse.raw.media_warnings`.
- **Групповой токен и фото (VK error 27).** VK-метод `photos.getWallUploadServer`/`photos.saveWallPhoto` **недоступен с групповым токеном** (`Group authorization failed`, `error_code=27`). Текстовая публикация (`wall.post`) с групповым токеном работает. Поэтому при `error_code=27` бот **не роняет публикацию**, а делает **text-only fallback**: постит текст без вложения и добавляет в `raw`:
  ```json
  {
    "media_upload_skipped": true,
    "media_upload_error_code": 27,
    "media_warnings": ["VK photo upload skipped: group token cannot call photos.getWallUploadServer/photos.saveWallPhoto"]
  }
  ```
  Любая **другая** ошибка upload flow остаётся `PublishError` (публикация помечается `failed`).
- **Полноценная загрузка фото через user-token** (или через отдельный сервис аплоада) — задача будущего этапа. Токен нигде не логируется и не попадает в `raw`/тексты ошибок.

## Безопасная live-публикация (флаги)

Реальная отправка в соцсети **по умолчанию выключена** — без явных флагов опубликовать ничего нельзя (защита от случайной публикации):

| Флаг (`.env`) | По умолчанию | Что включает |
|---------------|--------------|--------------|
| `TELEGRAM_LIVE_PUBLISHING_ENABLED` | `false` | Реальный вызов Telegram Bot API `sendMessage` |
| `VK_LIVE_PUBLISHING_ENABLED` | `false` | Реальный вызов VK `wall.post` |

- **Флаг `false`:** `publish` возвращает `PublishError` «Live publishing disabled by config», **сеть не вызывается**, публикация помечается `failed` с этим сообщением.
- **Флаг `true`:** Telegram использует `TELEGRAM_BOT_TOKEN` + `TELEGRAM_DEFAULT_CHANNEL_ID`, VK — `VK_ACCESS_TOKEN` + `VK_DEFAULT_GROUP_ID`. `GET /health/readiness` предупредит, если флаг включён, но токен/таргет не заданы.
- В тестах реальные HTTP-вызовы Telegram/VK всегда мокаются (`httpx.MockTransport`) — настоящая сеть не дёргается.

**Dry-run preview.** Можно посмотреть, что и куда ушло бы, **без отправки**:

- API: `POST /post-publications/preview/{post_id}` → платформы, `target_id`, текст, `preferred_media_path`, `media_source`, `media_kind`, `would_attach_media`, `live_enabled` (ничего не публикует, сеть не вызывается);
- CLI: `python -m app.scripts.publish_post --post-id 1 --dry-run`.

## PostPublication

Каждая публикация — строка `post_publications` (модель `PostPublication`): платформа, цель (`target_id`), статус, внешний id/url, время плана/публикации, ошибка, число попыток (`attempts`) и `payload` (текст, `media_asset_id`, attachment, raw-ответ). Подробности модели — в [`Докс/06_Модель_данных.md`](./06_Модель_данных.md) (раздел «PostPublication»).

## Идемпотентность

- Уникальный индекс `ix_post_publications_post_id_platform` (`post_id` + `platform`) — один пост не публикуется в одну платформу дважды.
- `upsert_publication_schedule` обновляет существующую запись, не плодя дубли; уже опубликованную (`published`) не перезаписывает.
- При публикации: если публикация уже `published` и `force=false` — она помечается `skipped` (не публикуется повторно). С `force=true` — публикуется заново (растёт `attempts`).

## Ошибки публикации

Ошибка одной платформы **не** роняет весь процесс: соответствующая публикация получает `status=failed` и `error_message`, остальные платформы обрабатываются как обычно. `Post.status` переходит в `published` только если успешны все платформы и нет ошибок. На уровне API `publish` возвращает 200 со счётчиками `published_count`/`failed_count`/`skipped_count` — частичные сбои видны в ответе, а не как 5xx.

## Планировщик

`publish_due_publications_job(db, service, now=None)` (`app/scheduler/jobs.py`) делает один проход: берёт созревшие публикации (`scheduled`/`pending` с `scheduled_at <= now` или без даты), группирует по посту и публикует. Без бесконечного цикла и без сети (клиенты подменяемы) — задача легко тестируется и переиспользуется из CLI.

## API и команды

REST API `/post-publications` (`backend/app/api/post_publications.py`):

- `GET /post-publications?post_id=&project_id=&platform=&status=&limit=&offset=` — список;
- `GET /post-publications/{publication_id}` — публикация по id (404);
- `POST /post-publications/schedule/{post_id}` — запланировать (404/409);
- `POST /post-publications/publish/{post_id}` — опубликовать (404/409; частичные сбои — в теле);
- `POST /post-publications/publish-due` — опубликовать созревшие (тело `{ "now": "..." }` опционально);
- `POST /post-publications/preview/{post_id}` — **dry-run preview** payload без отправки (404 — нет поста);
- `PATCH /post-publications/{publication_id}` — ручная правка (`target_id`/`status`/`error_message` и пр.).

CLI:

```bash
make schedule-post post_id=1
make publish-post post_id=1
make publish-due
```

## Медиа-вложения и фотоальбомы

Пост с группой медиа (несколько фото по общему тегу) публикуется одним постом с
несколькими фото — и в VK, и в Telegram:

- **VK** — фото загружаются через `photos.getWallUploadServer` → upload →
  `photos.saveWallPhoto`, вложения `photo{owner}_{id}` через запятую в `wall.post`
  (лимит `VK_MEDIA_GROUP_MAX_PHOTOS`, по умолчанию 5). Групповой токен без прав
  `photos.*` (error_code=27) → безопасный text-only фолбэк.
- **Telegram** — фотоальбом через `sendMediaGroup` (2–10 фото, caption только в
  первом элементе) или `sendPhoto` для одного фото (лимит
  `TELEGRAM_MEDIA_GROUP_MAX_PHOTOS`, по умолчанию 10).
- Общее: HEIC/HEIF конвертируется в JPEG **в памяти** (оригинал не
  перезаписывается); видео пока пропускается с предупреждением; байты берутся из
  локальной enhanced-копии или публичной папки Яндекс Диска. Загрузка медиа и
  конвертация — в общем модуле `app/integrations/media_attachments.py`.

Подробно: VK — [22_Группировка_медиа_и_VK_посты.md](22_Группировка_медиа_и_VK_посты.md),
Telegram — [23_Telegram_media_group.md](23_Telegram_media_group.md).

## Мультиплатформенность (capability-слой)

Возможности платформ (текст/фото/альбом/видео, лимиты, live-флаг) вынесены в
`app/integrations/platform_capabilities.py`. `route_media` решает, что уйдёт на
каждую платформу; dry-run/preview показывает `would_attach_media`,
`media_warnings`, `unsupported_media_reason`, `platform_capabilities`.

- **Live-ready:** VK, Telegram (текст + фото-группа).
- **Skeleton / dry-run:** Instagram (фото/carousel/reels), YouTube и RuTube (видео).
  Live возвращает `Live publishing for <platform> is not implemented yet`.

Подробно — [24_Мультиплатформенная_публикация_медиа.md](24_Мультиплатформенная_публикация_медиа.md).

## Ограничения

- **Live-ready — только Telegram и VK**; Instagram/YouTube/RuTube — adapter-скелеты
  (preview работает, live пока не реализован);
- **в тестах нет реальных API и сети** — `FakePublishingClient` или `httpx.MockTransport`;
- live-публикация выключена по умолчанию: без `*_LIVE_PUBLISHING_ENABLED` отправка невозможна;
- SEO-заполнение группы выключено по умолчанию: без `VK_GROUP_SETUP_LIVE_ENABLED` — только preview;
- **видео** пока не загружается ни в VK, ни в Telegram — пропускается с предупреждением;
- **аналитика эффективности постов** — Этап 8.
