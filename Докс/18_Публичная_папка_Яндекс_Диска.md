# Публичная папка Яндекс Диска (SMM)

Документ описывает альтернативный режим работы с медиа: чтение из **публичной папки** Яндекс Диска по public-ссылке. В этом режиме **OAuth-токен не требуется**. Приватный режим (через `YANDEX_DISK_TOKEN`) сохраняется и не меняется.

## Зачем это нужно

Раньше бот читал медиа только через приватный OAuth-токен. Теперь команда может просто расшарить папку SMM публичной ссылкой, и бот будет брать из неё метаданные файлов — без выдачи токенов.

## Структура папки SMM

Публичная ссылка ведёт на ресурс, внутри которого лежит папка `SMM`, а в ней — две проектные папки:

```
<публичная ссылка>/
└── SMM/
    ├── Тион/                  # медиа проекта TEEON
    │   ├── futbolki-logotip-shelkografiya.jpg
    │   └── hoodie-dtf-merch.mp4
    └── Фабрика сувениров/      # медиа проекта «Фабрика сувениров»
        ├── kruzhki-uf-pechat.jpg
        └── gravirovka-ezhednevnik.jpg
```

Имя корневой папки задаётся настройкой `YANDEX_DISK_PUBLIC_ROOT_FOLDER` (по умолчанию `SMM`). Если публичная ссылка ведёт прямо на папку `SMM`, бот это поймёт (есть фолбэк на корень).

## Правила доступа медиа по проектам (критично)

| Проект (slug) | Может брать из папок | НЕ может брать |
|---------------|----------------------|----------------|
| TEEON (`teeon`) | только «Тион» | «Фабрика сувениров» |
| Фабрика сувениров (`fabric-souvenirs`) | «Фабрика сувениров» **и** «Тион» | — |

- **TEEON никогда не использует медиа из «Фабрика сувениров».**
- **Фабрика сувениров** может использовать и свои медиа, и медиа из «Тион».

Правила живут в `app/services/project_media_paths.py`:

- `get_public_project_folder_names(slug)` — список разрешённых имён папок (с синонимами);
- `is_public_folder_allowed_for_project(slug, имя_или_путь)` — проверка доступа;
- `get_public_scan_roots(slug)` — канонические пути папок-источников.

Имена сравниваются **нормализованно** (нижний регистр, ё→е, без пробелов/дефисов/подчёркиваний, по последнему сегменту пути), поэтому «Тион», «TEEON», «teeon», «Tion» — одна и та же папка.

## Как называть файлы тегами

Файлы именуются по содержимому через дефисы — это помогает тегированию (`MediaTaggingService`):

- `futbolki-logotip-shelkografiya.jpg`
- `hoodie-dtf-merch.mp4`
- `kruzhki-uf-pechat.jpg`
- `gravirovka-ezhednevnik.jpg`

Из имени файла извлекаются теги (изделия, технологии, детали), заголовок (имя без расширения) и описание (источник — публичная папка SMM).

## Что бот делает (и чего НЕ делает)

- Берёт **только метаданные**: имя, путь, размер, mime, теги. **Файлы НЕ скачиваются** и не загружаются на сервер.
- Создаёт/обновляет `MediaAsset` с `source_type=internal`, `license_type=company_owned`, `status=new`, путём `public://yandex/<slug>/SMM/<папка>/<файл>`. Путь содержит slug проекта, поэтому один и тот же файл из общей папки «Тион» становится **отдельным** активом для teeon и для fabric-souvenirs (без коллизий и без перезаписи чужих строк).
- Пропускает файлы, которые не являются фото или видео.
- Идемпотентен: повторная синхронизация не плодит дубликаты (upsert по `yandex_disk_path`).
- Правило доступа проверяется по **полному пути** файла: если чужая проектная папка вложена в разрешённую (например, «Фабрика сувениров» внутри «Тион»), её файлы для teeon **не берутся** (`is_public_path_allowed_for_project`).

## Большие папки: пагинация

Яндекс отдаёт содержимое папки **страницами** (по умолчанию 100 элементов). В реальных папках бывают **сотни файлов** (например, в «Тион» 190+). Поэтому листинг идёт постранично:

- `YandexDiskPublicClient.list_all_public_resources(...)` собирает все страницы (с защитой `max_pages`, по умолчанию 50, и таймаутом `timeout=20с` на запрос — чтобы зависший ответ не блокировал sync);
- рекурсивный обход (`list_public_files_recursive`) и поиск файла (`find_public_resource_by_path`) используют именно её.

Поэтому `found_files` для большой папки может быть **больше 100** — это нормально.

## Устаревшие (stale) публичные медиа

Если файл на Диске **переименовали или удалили** после прошлой синхронизации, путь в БД (`MediaAsset.yandex_disk_path`) становится **устаревшим** — такой файл нельзя скачать (улучшение выдаст «Публичный ресурс не найден»). После каждой синхронизации бот сравнивает пути в БД с тем, что реально на диске, и если находит «осиротевшие» записи — добавляет **warning** в `result.errors`:

```
Stale public media not found on disk: N
  stale: public://yandex/teeon/teeon/IMG_5113.HEIC
  ...
```

Важно:

- stale-записи **НЕ удаляются** и **НЕ меняют статус** автоматически (чтобы не потерять историю/связи) — это лишь предупреждение;
- лечится повторной синхронизацией: **после переименования файлов на Диске запустите `make sync-public-media project_slug=<slug>`** — она создаст актуальные `MediaAsset` под текущие имена, после чего улучшение заработает;
- сетевые ошибки/таймаут одной папки не валят весь sync — они попадают в `result.errors`, остальные папки обрабатываются;
- stale-проверка делается **только при полностью успешном скане**: если хоть одна разрешённая папка не дочиталась (ошибка/таймаут), предупреждения о stale **не выдаются** — иначе файлы из недочитанной папки ложно помечались бы устаревшими.

## Как включить и проверить

1. В `.env`:
   ```dotenv
   YANDEX_DISK_PUBLIC_MODE=true
   YANDEX_DISK_PUBLIC_SMM_URL=https://disk.yandex.ru/d/PYnchGnSLKW3yw
   YANDEX_DISK_PUBLIC_ROOT_FOLDER=SMM
   ```
2. Синхронизация:
   ```bash
   make sync-public-media project_slug=teeon
   make media-summary project_slug=teeon

   make sync-public-media project_slug=fabric-souvenirs
   make media-summary project_slug=fabric-souvenirs
   ```
3. Проверка через API (без токена):
   ```bash
   curl -X POST http://localhost:8000/media-assets/sync/public/slug/teeon
   curl "http://localhost:8000/media-assets?project_id=1"
   ```
4. Готовность: `GET /health/readiness` показывает `yandex_disk_public_mode` (без раскрытия токенов).

## Что дальше: улучшение медиа

После публичной синхронизации одобренные изображения можно подготовить к
публикации — бот создаёт **улучшенные копии** локально (свет/контраст, ресайз,
конвертация), не трогая оригиналы и не перезаписывая файлы на Яндекс Диске. Именно
здесь файлы впервые **скачиваются** (по download-ссылке публичного API) — но только
для построения производной копии. Подробности — в
[`19_Улучшение_медиа.md`](./19_Улучшение_медиа.md):

```bash
make enhance-project-media project_slug=teeon
make media-enhancement-summary project_slug=teeon
```

## Ограничения

- Реальные стоки и сеть в тестах не вызываются — публичный клиент подменяется (`httpx.MockTransport` / fake-клиент).
- На этапе синхронизации файлы не скачиваются (только метаданные); скачивание происходит лишь при улучшении медиа (см. выше).
- Доступ строго по правилам выше: teeon не видит «Фабрика сувениров».
