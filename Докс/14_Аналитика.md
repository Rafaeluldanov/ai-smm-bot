# Аналитика (Этап 8)

Документ описывает backend-логику **аналитики публикаций**: хранение снимков метрик, расчёт эффективности и feedback-сигналы для будущей приоритизации тем. Реальные analytics API соцсетей и Instagram на этом этапе **не** подключены — метрики вводятся вручную (API/CLI) или берутся у детерминированного fake-провайдера.

## Какие метрики собираются

Каждый снимок (`PostAnalyticsSnapshot`) хранит метрики поста на одной платформе в момент `snapshot_at`:

- объём: `impressions` (показы), `reach` (охват), `views` (просмотры);
- вовлечения: `likes`, `reactions`, `comments`, `shares`, `saves`;
- переходы: `clicks`;
- расчётные: `ctr`, `engagement_rate`;
- `raw_metrics` — исходные данные «как пришли»; `source` — откуда (`manual`/`fake_provider`/`telegram_api`/`vk_api`).

Снимки накапливаются: по одному посту/платформе может быть несколько снимков (динамика во времени).

## Как считаются CTR и engagement_rate

Чистые функции в `backend/app/services/analytics_metrics.py` (без БД и сети, защита от деления на ноль, округление rate до 4 знаков):

- `engagements = likes + reactions + comments + shares + saves`;
- `ctr = clicks / max(impressions, reach, 1)`;
- `engagement_rate = engagements / max(reach, impressions, 1)`;
- `performance_score` (0..100) — взвешенная свёртка: объём охвата/показов (25), `engagement_rate` (35), `ctr` (25), абсолютные вовлечения (15).

`ctr` и `engagement_rate` рассчитываются и сохраняются в снимке при загрузке метрик.

## Как формируются отчёты

`AnalyticsService` (`backend/app/services/analytics_service.py`) агрегирует снимки:

- **PostPerformanceReport** (`get_post_performance`) — итоги по посту (суммы метрик, средние rate, разбивка по платформам).
- **TopicPerformanceReport** (`get_topic_performance`) — группировка снимков по `topic_id`, `performance_score`, сортировка по убыванию.
- **ClusterPerformanceReport** (`get_cluster_performance`) — группировка по `topic.cluster` (футболки, худи, шелкография, кружки, гравировка и т. п.).
- **ProjectAnalyticsSummary** (`get_project_summary`) — сводка проекта: число постов/опубликованных/снимков, итоги и топ тем/кластеров.

Загрузка метрик:

- `ingest_snapshot` — ручной снимок по посту (через API `POST /analytics/snapshots`);
- `ingest_for_publication` — снимок по публикации из переданных `metrics` (без метрик → ошибка 422);
- `fetch_and_store_for_publication` — метрики у fake-провайдера (`source=fake_provider`).

## Как работают feedback-сигналы

`build_feedback_signals` анализирует кластеры и формирует сигналы для будущей корректировки приоритетов тем:

- `boost_cluster` — высокий `performance_score` (> 60): усилить кластер;
- `review_content_format` — низкая вовлечённость (`engagement_rate` < 0.02): пересмотреть формат контента;
- `improve_cta` — много показов (≥ 1000) при низком CTR (< 0.01): улучшить призыв к действию;
- если данных нет — добавляется предупреждение.

`TopicAnalyticsFeedbackService` (`topic_analytics_feedback_service.py`) превращает это в практические корректировки: `build_business_priority_adjustments` возвращает `{cluster: ±10}` (score > 60 → +10, score < 25 → −10), `build_feedback_notes` — текстовые заметки. Это лёгкая интеграция, не меняющая Этап 4: на будущем этапе результат можно подмешивать в `TopicSelectionService.business_priorities`.

## API и команды

REST API `/analytics` (`backend/app/api/analytics.py`):

- `GET /analytics/snapshots?post_id=&project_id=&topic_id=&platform=` — список;
- `GET /analytics/snapshots/{snapshot_id}` — снимок по id (404);
- `POST /analytics/snapshots` — ручной снимок (404 — нет поста);
- `POST /analytics/ingest/publication/{publication_id}` — загрузка метрик по публикации (404/422);
- `POST /analytics/fetch/publication/{publication_id}` — метрики у fake-провайдера (404);
- `GET /analytics/posts/{post_id}/performance` — отчёт по посту (404);
- `GET /analytics/projects/{project_id}/{topics,clusters,summary,feedback}` — отчёты по проекту (404).

CLI:

```bash
make ingest-analytics post_id=1
make analytics-report project_slug=teeon
```

## Ограничения Этапа 8

- **реальные analytics API Telegram/VK не вызываются** — только ручной ввод и fake-провайдер;
- **Instagram не подключён**;
- **AI и внешние сетевые запросы не используются**;
- влияние на `priority_score` подготовлено (`TopicAnalyticsFeedbackService`), но в `TopicSelectionService` пока не вшито — это задел на будущее.
