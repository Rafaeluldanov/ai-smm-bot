# Botfleet SMM Platform Recommendations (v1.0.1)

Статическая экспертная база SMM-рекомендаций Botfleet: для каждой медиаплатформы — роль, частота,
сигналы алгоритмов, форматы, правила контента, риски, KPI, недельный ритм, чек-лист перед публикацией
и кросс-платформенная адаптация.

**Это НЕ новый AI-слой и НЕ автоматическое управление публикациями.** База полностью **read-only**:
она НЕ меняет расписание, НЕ создаёт и НЕ отправляет публикации, НЕ трогает автопостинг/CRM/бюджет,
НЕ запускает рекламу, НЕ создаёт workflow и НЕ ходит во внешние API.

## 1. Назначение

Дать SMM-специалисту и владельцу компактный экспертный ориентир по каждой площадке прямо в интерфейсе
Botfleet: что публиковать, как часто, на какие сигналы работает охват, чего нельзя делать, что мерить.
Рекомендации — практические ориентиры, а не гарантии платформ.

## 2. Versioning

Версия базы — поле `version` в ресурсе (текущая **`2026.1`**). Версия отображается в интерфейсе
(вкладка «Рекомендации» и общий экран) как «версия 2026.1». При обновлении данных — поднимайте
`version` (например, `2026.2`), чтобы отличать редакции. Имя файла содержит год (`..._2026.json`).

## 3. Структура JSON

Ресурс: `backend/app/resources/botfleet_smm_recommendations_2026.json` (UTF-8, валидный JSON).

Корневые ключи:
- `version`, `product` (= `Botfleet`), `language`, `title`, `disclaimer`;
- `universal_principles` — 8 универсальных принципов (title + points);
- `platforms` — объект `slug → {title, role, frequency_summary, frequency[], signals[], formats[],
  content_rules[], risks[], kpi[], extra_sections[]}`;
- `cross_platform_pipeline` — `{description, sources[{source, adaptations}], rules[], funnel}`;
- `weekly_rhythm` — `{note, time_guidelines, platforms{slug → {Пн..Вс}}}`;
- `pre_publish_checklist` — 8 пунктов.

Требования к ресурсу (проверяются тестами и `validate_knowledge_base`): UTF-8, валидный JSON, все 10
каналов, уникальные slug, версия заполнена, **нет HTML/скриптов** (никаких `<`/`>` в значениях),
**нет секретоподобных ключей**, `product = Botfleet` (без посторонних брендов в примерах).

## 4. Список платформ (canonical slug базы знаний)

`instagram`, `telegram`, `vk`, `youtube`, `rutube`, `dzen`, `ok`, `website`, `2gis`, `email`.

Сервис поддерживает **алиасы** (только внутри сервиса; canonical slug каталога проекта не меняются) —
в т. ч. фактические slug каталога Botfleet: `odnoklassniki → ok`, `two_gis → 2gis`; а также
`vkontakte/vk_com → vk`, `zen/yandex_dzen → dzen`, `site/web → website`, `gis2 → 2gis`,
`email_marketing → email`. Так вкладка на странице площадки (slug каталога `odnoklassniki`/`two_gis`)
корректно находит запись базы.

## 5. Сервис

`app/services/platform_recommendations_service.py` → `PlatformRecommendationsService`
(фабрика `get_platform_recommendations_service()`), stateless, read-only:
- `load_knowledge_base()` — читает локальный JSON (кэш `lru_cache`, возвращает копии);
- `normalize_platform_slug()` — whitelist `[a-z0-9_]` + алиасы → canonical slug (неизвестная → 404);
- `list_platforms()` — краткий список `{slug, title, role, frequency_summary}`;
- `get_platform_recommendations(slug)` — полные рекомендации по платформе;
- `get_universal_recommendations()` — принципы/конвейер/ритм/чек-лист;
- `validate_knowledge_base()` — целостность базы (каналы/секции/типы/slug/секреты/HTML/версия).

Ошибки: `PlatformRecommendationsError` (битый ресурс → контролируемый 500), `UnknownPlatformError`
(неизвестная платформа → 404).

## 6. API (read-only)

- `GET /platform-recommendations` — список платформ;
- `GET /platform-recommendations/universal` — универсальные рекомендации;
- `GET /platform-recommendations/{platform_slug}` — рекомендации по платформе (canonical/alias);
- `GET /projects/{project_id}/platforms/{platform_slug}/recommendations` — то же под **tenant-гардом**
  (`require_project_access`), т. к. страницы платформ project-scoped.

Никаких мутаций БД, UsageEvent, платежей, audit на GET, внешних вызовов. Неизвестная платформа → 404;
битый ресурс → 500 без stack trace/путей в ответе.

## 7. Вкладка «Рекомендации»

Добавлена на страницу платформы `/ui/projects/{project_id}/platforms/{platform}` (функция
`ui_platform_workspace`): к списку вкладок добавляется `Рекомендации`, если у платформы (после
нормализации slug) есть запись в базе. Пане рендерится **на сервере** и **полностью экранируется**
(`html.escape`) — XSS невозможен. Карточки: Роль платформы · Частота (заметный блок) · Что публиковать
· Как работает охват · Правила контента · Что нельзя делать · Что измерять (KPI, списком) · План
недели (компактная таблица) · Перед публикацией (визуальные checkbox, состояние **не пишется в БД**) ·
Кросс-платформенная адаптация · Дополнительно · Универсальные принципы. Вверху — версия базы и
disclaimer. Вкладка **не меняет настройки площадки и не запускает автопостинг**.

## 8. Общий экран

`GET /ui/recommendations` — «SMM-рекомендации Botfleet»: универсальные принципы, сводная частота,
недельный календарь (все платформы × дни), кросс-платформенный конвейер, чек-лист и карточки всех
платформ (раскрываются на месте). Сайт/2ГИС/Email показаны как **дополнительные каналы продвижения**
(отдельной интеграции для них не создаётся). Ссылка на экран есть в разделе «Гайд».

## 9. Disclaimer

В интерфейсе показывается пометка: «Практические рекомендации Botfleet, версия 2026. Алгоритмы
платформ меняются. Проверяйте выводы на собственной статистике.»

## 10. Практические ориентиры vs гарантии

Точные проценты (штрафы охвата ВК, лимит репостов Instagram, CTR/дочитывания Дзен, open rate Email)
поданы как **«практический ориентир» / «практическое наблюдение»**, а НЕ как официальная гарантия
платформы. Формулировка «официально подтверждено платформой» в базе **не используется**. Тезис
Telegram про доставку подан как практический, не как гарантия просмотра каждым подписчиком.

## 11. Почему рекомендации не меняют автопостинг

База — справочное знание. Она физически не связана с движком расписаний/публикаций: сервис и API
только читают статический JSON, не имеют доступа к БД-мутациям, не вызывают публикацию и не трогают
`autopilot`/`schedule`/`live`. Автопостинг остаётся под своими существующими правилами и ревью.

## 12. Как обновлять рекомендации

1. Отредактируйте `backend/app/resources/botfleet_smm_recommendations_2026.json` (или создайте новый
   файл следующей редакции и обновите `_RESOURCE_PATH`).
2. Поднимите `version` (например, `2026.2`).
3. Не добавляйте HTML/теги (`<`, `>`) и секретоподобные ключи — валидатор их отклонит.
4. Прогоните проверку новой версии (см. ниже) и тесты содержимого.

## 13. Как проверить новую версию базы

```python
from app.services.platform_recommendations_service import get_platform_recommendations_service
get_platform_recommendations_service().validate_knowledge_base()  # True или PlatformRecommendationsError
```
Плюс автотесты:
`pytest tests/test_platform_recommendations_resource.py tests/test_platform_recommendations_content.py`.
Кэш `lru_cache` живёт в процессе — при ручной проверке перезапустите процесс/воркер.

## 14. Как откатить обновление ресурса

Ресурс версионируется в git вместе с кодом. Откат — обычный git-откат файла ресурса:
`git checkout <прошлый_коммит> -- backend/app/resources/botfleet_smm_recommendations_2026.json`
(или восстановите предыдущий файл редакции и верните `_RESOURCE_PATH`). Миграций БД нет — откат
безопасен и не затрагивает данные.

## 15. Что нужно обновлять вручную при изменении алгоритмов платформ

Всё содержимое — экспертное и статическое, поэтому при изменении алгоритмов площадок вручную
обновляйте: частоты, сигналы, правила, риски (включая проценты-ориентиры), KPI, недельный ритм,
кросс-платформенный конвейер и disclaimer/версию. Botfleet сам эти данные не собирает и не
верифицирует — источник истины — редактор базы знаний.
