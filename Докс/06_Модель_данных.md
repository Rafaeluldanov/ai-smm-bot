# Модель данных

Документ описывает сущности базы данных AI-SMM-бота, их назначение, поля, статусы и связи. Модели реализованы через SQLAlchemy, миграции — через Alembic, СУБД — PostgreSQL. Все поля-коллекции (`tags`, `seo_keywords`, `hashtags`) хранятся как JSON и в PostgreSQL используют тип **JSONB** (бинарное представление с поддержкой индексов и операторов по содержимому).

Общие соглашения:

- `id` — целочисленный первичный ключ (autoincrement).
- `created_at`, `updated_at` — временные метки (UTC), `created_at` проставляется при вставке, `updated_at` обновляется при каждом изменении.
- Внешние ключи именуются по шаблону `<сущность>_id` и ссылаются на `id` родительской таблицы.

---

## Project

**Назначение:** проект компании, для которого бот ведёт соцсети (например, TEEON или «Фабрика сувениров»). Корневая сущность: к проекту привязаны медиа, темы и публикации.

**Таблица:** `projects`

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | int (PK) | Уникальный идентификатор проекта |
| `name` | str | Человекочитаемое название (например, «TEEON») |
| `slug` | str (unique) | Машинное имя для URL/папок/настроек (например, `teeon`); уникален в рамках всей таблицы |
| `description` | str \| null | Краткое описание направления деятельности |
| `website_url` | str \| null | Адрес сайта проекта (например, `https://teeon.ru`) |
| `is_active` | bool | Флаг активности (по умолчанию `true`); неактивные проекты исключаются из автоматики |
| `created_at` | datetime | Дата создания записи |
| `updated_at` | datetime | Дата последнего изменения |

**Уникальность `slug`:** на поле `slug` наложен уникальный индекс `ix_projects_slug` — двух проектов с одинаковым `slug` в базе быть не может. Попытка создать или обновить проект с уже занятым `slug` приводит к ошибке API `409 Conflict`.

**Правила валидации `slug`:**

- допустимы только латинские буквы, цифры, дефис `-` и подчёркивание `_`;
- значение нормализуется к нижнему регистру (например, `TEEON` → `teeon`);
- пробелы и кириллица не допускаются (запрос отклоняется с `422 Unprocessable Entity`);
- минимальная длина — 2 символа.

**Деактивация вместо удаления:** проекты не удаляются из базы. Операция деактивации лишь выставляет `is_active = false`, сохраняя запись и все связанные с ней данные. По умолчанию список проектов отдаёт только активные записи (фильтр `active_only=true`).

**Связи:** одна запись Project ссылается на множество MediaAsset, Topic и Post (через `project_id`). Внешние ключи `project_id` настроены с правилом **`ON DELETE CASCADE`**: при (теоретическом) удалении проекта связанные медиа, темы и посты удаляются вместе с ним. В таблице Post внешние ключи `topic_id` и `media_asset_id` настроены с правилом **`ON DELETE SET NULL`**: при удалении темы или медиа соответствующая ссылка в посте обнуляется, а сам пост сохраняется.

## MediaAsset

**Назначение:** единица медиа из хранилища (Яндекс Диск) — фото или видео изделия. Бот сканирует Диск, разбирает имя файла, анализирует изображение и сохраняет метаданные с тегами для последующего подбора в посты.

**Таблица:** `media_assets`

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | int (PK) | Уникальный идентификатор медиа |
| `project_id` | int (FK → `projects.id`) | Проект, к которому относится файл |
| `file_name` | str | Имя файла как на Диске (используется при разборе тегов) |
| `yandex_disk_path` | str (unique) | Полный путь к файлу на Яндекс Диске; бизнес-уникален (см. ниже) — служит ключом anti-duplicate при синхронизации |
| `source_type` | str | Источник медиа: `internal` (собственные файлы компании) или `external_stock` (внешние/стоковые картинки); индексирован |
| `license_type` | str \| null | Тип лицензии/прав: `company_owned`, `external_needs_review`, `commercial_use_required`; критичен для внешних картинок |
| `title` | str \| null | Краткий заголовок медиа |
| `description` | str \| null | Подробное описание (в т.ч. результат анализа изображения) |
| `tags` | JSON/JSONB | Структурированные теги (см. ниже) |
| `status` | str | Статус обработки/готовности (см. ниже); индексирован |
| `last_used_at` | datetime \| null | Когда медиа последний раз использовалось в посте |
| `created_at` | datetime | Дата создания записи |
| `updated_at` | datetime | Дата последнего изменения |

**Статусы (`status`):**

| Значение | Смысл |
|----------|-------|
| `new` | Обнаружен на Диске (папка «Входящие на разбор»), ещё не одобрен |
| `approved` | Одобренное фото, готово к использованию в публикациях |
| `approved_video` | Одобренное видео, готово к использованию в публикациях |
| `needs_license_review` | Внешняя картинка — требуется проверка прав/лицензии перед использованием |
| `needs_reshoot` | Кадр требует досъёмки/переснятия (папка «06_Нужно_переснять» или ручная пометка); пока не годен для публикаций |
| `rejected` | Медиа отклонено и не используется (брак, дубликат, неподходящий кадр, отказ по лицензии) |
| `used` | Медиа уже использовалось в опубликованном посте |

Допустимый набор статусов и правила их смены централизованы в `MediaStatusService` (Этап 3): функции `get_allowed_statuses`, `can_transition`, `validate_transition`, `update_media_status`. Неизвестный статус вызывает `InvalidMediaStatusError` (API → `422`), запрещённый переход — `InvalidMediaStatusTransitionError` (API → `409`).

**Переходы статусов:**

Не любой статус можно сменить на любой другой — разрешён лишь ограниченный набор переходов. Таблица допустимых переходов (Этап 3):

| Из статуса | Допустимые целевые статусы |
|------------|----------------------------|
| `new` | `approved`, `approved_video`, `needs_license_review`, `rejected`, `needs_reshoot` |
| `approved` | `used`, `rejected`, `needs_reshoot` |
| `approved_video` | `used`, `rejected`, `needs_reshoot` |
| `needs_license_review` | `approved`, `rejected` |
| `needs_reshoot` | `approved`, `rejected` |
| `used` | `approved`, `approved_video` |
| `rejected` | `approved`, `needs_reshoot` |

Пояснения:

- из `new` файл расходится по любому из исходов разбора: одобрение фото/видео, отправка на проверку лицензии, отклонение или пометка на досъёмку;
- одобренные фото и видео (`approved`, `approved_video`) могут уйти в `used` после публикации, быть отклонены или отправлены на досъёмку;
- из проверки лицензии (`needs_license_review`) путь только в `approved` (права подтверждены) или `rejected` (права не подходят);
- кадр на досъёмке (`needs_reshoot`) после переснятия одобряется (`approved`) либо отклоняется (`rejected`);
- использованное медиа (`used`) можно вернуть в `approved`/`approved_video` для повторного применения;
- отклонённое (`rejected`) может быть реабилитировано в `approved` или отправлено на досъёмку (`needs_reshoot`).

Переходы вида «статус в самого себя» в таблице не перечислены и не разрешены.

**Индексы Этапа 2:** на таблицу наложены индексы `ix_media_assets_yandex_disk_path` (**уникальный** — гарантирует бизнес-уникальность пути), `ix_media_assets_status` и `ix_media_assets_source_type` (миграция `0002`). Уникальность `yandex_disk_path` используется при синхронизации как ключ upsert: повторное сканирование того же файла не создаёт дубликат, а обновляет существующую запись (или оставляет без изменений).

**Связи:** MediaAsset принадлежит одному Project (`project_id`) и может быть использован в ноль или нескольких Post (через `Post.media_asset_id`).

### Структура поля `tags`

Структура БД на Этапе 3 **не менялась** — поле `tags` по-прежнему хранит JSON (JSONB в PostgreSQL), но его содержимое стало богаче. Начиная с Этапа 3 `tags` — это JSON-объект со следующими ключами:

```json
{
  "products": ["..."],
  "technologies": ["..."],
  "details": ["..."],
  "materials": ["..."],
  "colors": ["..."],
  "categories": ["..."],
  "use_cases": ["..."],
  "audiences": ["..."],
  "topics": ["..."],
  "seo_keywords": ["..."],
  "matched_terms": ["..."],
  "confidence": 0.0,
  "needs_review": false,
  "review_reasons": ["..."]
}
```

Смысл ключей:

- `products` — изделия на фото (худи, футболка, кружка, ручка, шоппер и т.п.);
- `technologies` — технологии нанесения/изготовления (шелкография, DTF, DTG, УФ-печать, тампопечать, гравировка, вышивка и т.п.);
- `details` — конструктивные детали и особенности (карман, манжет, капюшон, а также жаккард, бирка, шеврон — отнесены к деталям);
- `materials` — материалы (хлопок, флис, керамика, металл и т.п.);
- `colors` — цвета изделия/нанесения;
- `categories` — категории медиа (см. перечень ниже): `apparel`, `souvenirs`, `branding`, `production`, `packaging`, `event_merch`, `external_reference`;
- `use_cases` — сценарии применения/назначение (корпоративный подарок, промо, форма и т.п.);
- `audiences` — целевые аудитории (b2b, сотрудники, мероприятие и т.п.);
- `topics` — предлагаемые темы публикаций, выведенные из содержимого файла;
- `seo_keywords` — SEO-запросы, выведенные из тегов (для подсказок при подготовке постов);
- `matched_terms` — список терминов из словарей таксономии, которые удалось сопоставить с именем/путём файла (для прозрачности и отладки тегирования);
- `confidence` — число `0.0–1.0`, уверенность тегирования (чем больше распознано значимых терминов, тем выше);
- `needs_review` — булев флаг: нужно ли ручное ревью результата тегирования;
- `review_reasons` — список строк с причинами, по которым выставлен `needs_review` (например, низкая уверенность, отсутствие изделий и технологий, внешний источник).

Флаг `needs_review` выставляется в `true`, если выполнено хотя бы одно из условий: `confidence < 0.35`; не распознано ни одного изделия (`products`) и ни одной технологии (`technologies`); `source_type == external_stock`; либо путь файла содержит `04_Внешние_картинки_из_интернета`. Соответствующие причины попадают в `review_reasons`.

**Категории (`categories`):**

| Значение | Смысл |
|----------|-------|
| `apparel` | На фото изделие одежды (худи, футболка и т.п.) |
| `souvenirs` | Сувенирный продукт (кружка, ручка, пакет и т.п.) |
| `branding` | Присутствует технология нанесения/брендирования |
| `production` | Производственный контекст (цех, производство, станок, процесс) |
| `packaging` | Упаковка (коробка, пакет, упаковка) |
| `event_merch` | Мерч под мероприятие/выставку/конференцию/event |
| `external_reference` | Внешний референс (`source_type == external_stock` или путь внешних картинок) |

На Этапе 3 поле `tags` заполняется сервисом `MediaTaggingService` (метод `analyze_file_name(file_name, project_slug, yandex_disk_path, source_type)`) — он опирается на словари таксономии из `media_taxonomy.py` и учитывает морфологию (точное совпадение токена, совпадение по префиксу, по общему префиксу с расхождением хвоста ≤2 символов; многословные термины — как подстрока; короткие коды вроде `dtf`, `dtg`, `uv`, `hr`, `b2b` — только точное совпадение). Для совместимости сохранён прежний метод `extract_tags_from_file_name`. Заполнение происходит при синхронизации с Диском, а также может быть пересчитано по данным БД через ретегирование (`MediaAnalysisService`, эндпоинты `retag/*`, CLI `retag_media.py`, `make retag-media`) — без обращения к Диску.

**Пример** для файла `Худи с карманом с шелкографией и жаккардами.jpg` (проект TEEON):

```json
{
  "products": ["худи"],
  "technologies": ["шелкография"],
  "details": ["карман", "жаккард"],
  "materials": [],
  "colors": [],
  "categories": ["apparel", "branding"],
  "use_cases": [],
  "audiences": [],
  "topics": ["худи с логотипом", "корпоративный мерч", "шелкография на худи"],
  "seo_keywords": ["худи с логотипом", "шелкография на одежде"],
  "matched_terms": ["худи", "карман", "шелкография", "жаккард"],
  "confidence": 0.7,
  "needs_review": false,
  "review_reasons": []
}
```

### Классификация по папкам Яндекс Диска (Этап 2)

При синхронизации (`YandexDiskMediaSyncService`) каждый найденный файл классифицируется по папке, в которой он лежит. От папки зависят `source_type`, `license_type` и `status` новой записи:

| Папка на Диске | `source_type` | `license_type` | `status` |
|----------------|---------------|----------------|----------|
| `01_Входящие_на_разбор` | `internal` | `company_owned` | `new` |
| `02_Одобренные_фото` | `internal` | `company_owned` | `approved` |
| `03_Видео` | `internal` | `company_owned` | `approved_video` |
| `04_Внешние_картинки_из_интернета` | `external_stock` | `external_needs_review` | `needs_license_review` |
| `06_Нужно_переснять` | `internal` | `company_owned` | `needs_reshoot` |

Папки разворачиваются от корня проекта на Диске (`/SMM_BOT/<папка_проекта>`). На Этапе 3 список сканируемых папок (`get_default_scan_folders`) расширен до пяти — добавлена `06_Нужно_переснять`. Папка `05_Использовано_в_постах` **не сканируется**. Файлы из любой иной локации в рамках корня по умолчанию трактуются как `internal` / `company_owned` / `new` (как «Входящие»). При синхронизации в `MediaTaggingService` передаются `project_slug`, `yandex_disk_path` и `source_type`, поэтому теги получаются богаче, чем при разборе одного только имени файла.

## Topic

**Назначение:** тема для будущей публикации. Бот сам формирует темы в рамках заданных человеком стратегических направлений, опираясь на SEO-запросы, тренды, сезонность, конкурентов, наличие медиа и статистику прошлых постов.

**Таблица:** `topics`

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | int (PK) | Уникальный идентификатор темы |
| `project_id` | int (FK → `projects.id`) | Проект, к которому относится тема |
| `title` | str | Формулировка темы (например, «худи с логотипом») |
| `cluster` | str \| null | Семантический кластер/группа тем |
| `priority_score` | float \| null | Расчётный приоритет (тренды, SEO, наличие фото и т.д.) |
| `business_priority` | int \| null | Бизнес-приоритет, заданный человеком (направление месяца) |
| `seo_keywords` | JSON/JSONB | Массив строк — ключевые запросы (см. ниже) |
| `status` | str | Статус темы (см. ниже) |
| `created_at` | datetime | Дата создания записи |
| `updated_at` | datetime | Дата последнего изменения |

**Статусы (`status`):**

| Значение | Смысл |
|----------|-------|
| `proposed` | Предложена ботом, ожидает обработки |
| `approved` | Одобрена для генерации постов |
| `rejected` | Отклонена, не используется |
| `used` | По теме уже сгенерированы публикации |

**Связи:** Topic принадлежит одному Project (`project_id`) и порождает ноль или несколько Post (через `Post.topic_id`).

## Post

**Назначение:** публикация для соцсетей. Содержит отдельные тексты под каждую площадку, привязанное медиа, хэштеги и SEO-ключи. Проходит путь от черновика через согласование к публикации.

**Таблица:** `posts`

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | int (PK) | Уникальный идентификатор поста |
| `project_id` | int (FK → `projects.id`) | Проект публикации |
| `topic_id` | int (FK → `topics.id`) | Тема, на основе которой создан пост |
| `media_asset_id` | int (FK → `media_assets.id`) \| null | Прикреплённое медиа (опционально) |
| `title` | str \| null | Внутренний заголовок/название черновика |
| `telegram_text` | str \| null | Текст для Telegram |
| `vk_text` | str \| null | Текст для VK |
| `instagram_text` | str \| null | Текст для Instagram |
| `hashtags` | JSON/JSONB | Массив строк — хэштеги (см. ниже) |
| `seo_keywords` | JSON/JSONB | Массив строк — SEO-ключи (см. ниже) |
| `status` | str | Статус поста (см. ниже) |
| `scheduled_at` | datetime \| null | Запланированное время публикации |
| `published_at` | datetime \| null | Фактическое время публикации |
| `created_at` | datetime | Дата создания записи |
| `updated_at` | datetime | Дата последнего изменения |

**Статусы (`status`):**

| Значение | Смысл |
|----------|-------|
| `draft` | Черновик сгенерирован ботом (медиа подобрано) |
| `needs_media` | Черновик создан, но подходящего медиа нет — требуется фото/видео |
| `needs_review` | Отправлен на согласование человеку (Этап 6) |
| `approved` | Согласован, готов к публикации |
| `scheduled` | Запланирован на `scheduled_at` |
| `published` | Опубликован (`published_at` заполнено) |
| `rejected` | Отклонён при согласовании |

**Связи:** Post принадлежит одному Project (`project_id`) и одной Topic (`topic_id`); опционально ссылается на один MediaAsset (`media_asset_id`).

### Поля `hashtags` и `seo_keywords`

Оба поля — это JSON-массивы строк. В PostgreSQL хранятся как JSONB.

```json
// hashtags
["#худи", "#корпоративныймерч", "#шелкография"]

// seo_keywords
["худи с логотипом", "корпоративный мерч на заказ", "шелкография на одежде"]
```

То же относится к `Topic.seo_keywords` — это массив строк (поисковых запросов).

## Связи между сущностями

| Связь | Кардинальность | Внешний ключ | Правило при удалении |
|-------|----------------|--------------|----------------------|
| Project → MediaAsset | 1 — N | `media_assets.project_id` | `ON DELETE CASCADE` |
| Project → Topic | 1 — N | `topics.project_id` | `ON DELETE CASCADE` |
| Project → Post | 1 — N | `posts.project_id` | `ON DELETE CASCADE` |
| Topic → Post | 1 — N | `posts.topic_id` | `ON DELETE SET NULL` |
| MediaAsset → Post | 0..1 — N | `posts.media_asset_id` (nullable) | `ON DELETE SET NULL` |

Схематично:

```
Project ──1:N── MediaAsset ──0..1─┐
   │                              │
   ├──1:N── Topic ──1:N── Post ───┘
   └──────────1:N──────── Post
```

Пояснения:

- **Project 1 — N MediaAsset / Topic / Post.** Один проект владеет множеством медиа, тем и постов; каждая из этих записей всегда принадлежит ровно одному проекту.
- **Topic 1 — N Post.** По одной теме может быть создано несколько публикаций (например, под разные площадки или в разные периоды).
- **MediaAsset 0..1 в Post.** У поста может быть привязано одно медиа или ни одного (`media_asset_id` допускает `null`); одно и то же медиа может переиспользоваться в разных постах.

На Этапе 5 `Topic` превращается в `Post` — черновик с отдельными текстами под Telegram, VK и Instagram, хэштегами, SEO-ключами и (по возможности) подобранным медиа. Реальный AI и публикация не задействованы.

## PostReviewAction — журнал согласования (Этап 6)

**Назначение:** журнал (audit log) действий по посту в процессе согласования. Каждое действие — отдельная неизменяемая запись; по ним строится таймлайн поста. Записи не редактируются и не удаляются.

**Таблица:** `post_review_actions`

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | int (PK) | Идентификатор действия |
| `post_id` | int (FK → `posts.id`) | Пост, к которому относится действие (`ON DELETE CASCADE`) |
| `action` | str | Тип действия (см. ниже); индексирован |
| `from_status` | str \| null | Статус поста до действия |
| `to_status` | str \| null | Статус поста после действия |
| `comment` | str \| null | Комментарий согласующего / причина отклонения |
| `actor_name` | str \| null | Кто выполнил действие |
| `actor_role` | str \| null | Роль исполнителя (например, `manager`) |
| `payload` | JSON/JSONB | Детали действия (изменённые поля, версии текста до/после) |
| `created_at` | datetime | Когда выполнено действие; индексирован |
| `updated_at` | datetime | Дата последнего изменения записи |

**Действия (`action`):**

| Значение | Смысл |
|----------|-------|
| `submit_for_review` | Черновик отправлен на согласование (`→ needs_review`) |
| `approve` | Пост одобрен (`→ approved`) |
| `reject` | Пост отклонён (`→ rejected`) |
| `request_changes` | Запрошена доработка (`→ draft`) |
| `return_to_draft` | Пост возвращён в черновик (`→ draft`) |
| `edit_text` | Ручная правка текстов поста |
| `change_media` | Изменено только прикреплённое медиа |
| `comment` | Комментарий без смены статуса |

**`payload`** — JSON-объект с деталями: для правок — `changed_fields`, `before`, `after` (версии до/после); для решений может хранить причину/комментарий. Для смены статуса значимы поля `from_status`/`to_status`.

**Индексы:** миграция `0005_post_review_actions` создаёт `ix_post_review_actions_post_id`, `ix_post_review_actions_action`, `ix_post_review_actions_created_at`.

**Связи:** PostReviewAction принадлежит одному Post (`post_id`, `ON DELETE CASCADE` — при удалении поста журнал удаляется вместе с ним). Смена статуса всегда проходит через `post_status_service.validate_transition` (см. таблицу переходов Post выше).

## PostPublication — публикация поста на платформе (Этап 7)

**Назначение:** одна запись = публикация одного поста на одной платформе (`telegram`/`vk`). Хранит цель публикации, статус, внешний идентификатор и ошибку. Бизнес-уникальность — пара (`post_id`, `platform`): один пост не публикуется в одну платформу дважды (идемпотентность).

**Таблица:** `post_publications`

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | int (PK) | Идентификатор публикации |
| `post_id` | int (FK → `posts.id`) | Пост (`ON DELETE CASCADE`); индексирован |
| `project_id` | int (FK → `projects.id`) | Проект (`ON DELETE CASCADE`); индексирован |
| `platform` | str | Платформа: `telegram` или `vk`; индексирован |
| `target_id` | str \| null | Канал Telegram / `owner_id` VK (куда публикуем) |
| `status` | str | Статус публикации (см. ниже); индексирован |
| `external_post_id` | str \| null | Идентификатор записи на платформе после публикации |
| `external_url` | str \| null | Ссылка на опубликованную запись |
| `scheduled_at` | datetime \| null | Запланированное время публикации; индексирован |
| `published_at` | datetime \| null | Фактическое время публикации; индексирован |
| `error_message` | str \| null | Текст ошибки при неудачной публикации |
| `attempts` | int | Число попыток публикации |
| `payload` | JSON/JSONB | Текст, `media_asset_id`, attachment, raw-ответ клиента, детали идемпотентности |
| `created_at` | datetime | Дата создания записи |
| `updated_at` | datetime | Дата последнего изменения |

**Статусы (`status`):**

| Значение | Смысл |
|----------|-------|
| `pending` | Готова к публикации, время не задано (публиковать сразу) |
| `scheduled` | Запланирована на `scheduled_at` |
| `publishing` | В процессе отправки на платформу |
| `published` | Опубликована (`external_post_id`/`published_at` заполнены) |
| `failed` | Публикация не удалась (`error_message` заполнено) |
| `skipped` | Пропущена (уже опубликована, без `force`) |

**Идемпотентность:** уникальный индекс `ix_post_publications_post_id_platform` (`post_id` + `platform`) и upsert по этой паре гарантируют, что повторное планирование/публикация не плодит дубли. Уже опубликованную запись повторная публикация без `force` помечает `skipped`.

**Индексы:** миграция `0006_post_publications` создаёт `ix_post_publications_post_id`, `ix_post_publications_project_id`, `ix_post_publications_platform`, `ix_post_publications_status`, `ix_post_publications_scheduled_at`, `ix_post_publications_published_at` и уникальный `ix_post_publications_post_id_platform`.

**Связи:** PostPublication принадлежит одному Post и одному Project. При публикации всех платформ без ошибок `Post.status` переходит в `published` (через `scheduled`), а `Post.published_at` проставляется.

**Назначение:** снимок метрик поста на одной платформе в момент времени. Снимки накапливаются; по ним считаются эффективность поста/темы/кластера/проекта и feedback-сигналы. Реальные API соцсетей не вызываются — метрики вводятся вручную или берутся у fake-провайдера.

**Таблица:** `post_analytics_snapshots`

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | int (PK) | Идентификатор снимка |
| `post_id` | int (FK → `posts.id`) | Пост (`ON DELETE CASCADE`); индексирован |
| `post_publication_id` | int \| null (FK → `post_publications.id`) | Публикация (`ON DELETE SET NULL`); индексирован |

| `topic_id` | int \| null (FK → `topics.id`) | Тема (`ON DELETE SET NULL`); индексирован |
| `platform` | str | `telegram` / `vk` / `manual` / `unknown`; индексирован |
| `snapshot_at` | datetime | Момент снимка; индексирован |
| `impressions` `reach` `views` | int | Показы, охват, просмотры |

Прогон не меняет правила отдельных сервисов: он лишь вызывает их (выбор тем, генерация, согласование, планирование, публикация) под контролем `AutonomousSafetyService`. **Индексы:** миграция `0009_autonomous_runs` — для `autonomous_runs`: `project_id`, `mode`, `status`, `created_at`; для `autonomous_run_steps`: `run_id`, `step_name`, `status`, `entity_type`, `entity_id`.

## MediaAssetVariant — улучшенная копия медиа (Этап: улучшение медиа)

**Назначение:** производная (улучшенная) КОПИЯ изображения. Оригинальный `MediaAsset` **никогда не изменяется и не перезаписывается** — улучшение всегда создаёт новый файл и отдельную запись `MediaAssetVariant` с метаданными обработки.

### MediaAssetVariant (`media_asset_variants`)

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | int (PK) | Идентификатор варианта |
| `media_asset_id` | int (FK → `media_assets.id`) | Исходный медиа-актив (`ON DELETE CASCADE`); индексирован |
| `project_id` | int (FK → `projects.id`) | Проект (`ON DELETE CASCADE`); индексирован |
| `variant_type` | str | `enhanced` / `social_preview` / `retouch_candidate`; индексирован |
| `status` | str | `created` / `needs_review` / `approved` / `rejected` / `failed`; индексирован |
| `source_media_asset_id` | int \| null (FK → `media_assets.id`) | Источник копии (`ON DELETE SET NULL`) |
| `source_path` | str \| null | Путь исходника (`yandex_disk_path` оригинала) |
| `output_path` | str \| null | Путь сохранённой копии в `MEDIA_ENHANCEMENT_STORAGE_DIR` |
| `output_format` | str \| null | Формат вывода (`jpg` / `png` / `webp`) |
| `width` `height` `file_size` | int \| null | Размеры и вес копии |
| `operations` | JSON | Список применённых операций (например, `["convert","resize","auto_contrast","sharpen"]`) |
| `before_metadata` `after_metadata` | JSON | Снимок параметров изображения до и после обработки |
| `quality_score` | float \| null | Грубая эвристика качества [0..1] |
| `warnings` | JSON | Предупреждения (из-за них правка может уйти в `needs_review`) |
| `error_message` | str \| null | Текст ошибки при `failed` |
| `created_at` / `updated_at` | datetime | Служебные метки |

**Связи:** один `MediaAsset` → много `MediaAssetVariant`. Вариант принадлежит проекту и трассируется к исходному активу (`media_asset_id` / `source_media_asset_id`). Удаление медиа/проекта каскадно удаляет варианты.

**Безопасность:** оригинал не трогается (ни путь, ни статус, ни файл). Видео не улучшаются. Спорные правки (баланс белого, denoise — меняют реальный цвет/текстуру изделия) → `status=needs_review`. Реальная AI-ретушь не подключена (`app/ai/image_editing.py` — заглушка).

**Индексы:** миграция `0010_media_asset_variants` — `media_asset_id`, `project_id`, `status`, `variant_type`.
