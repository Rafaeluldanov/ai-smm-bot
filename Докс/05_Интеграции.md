# Интеграции

В этом документе описаны все внешние интеграции AI-SMM-бота: их назначение, что требуется для подключения (включая имя переменной окружения для токена), на каком этапе разработки они подключаются и где в репозитории находится соответствующий клиент.

## Общие принципы

Перед детальным описанием — несколько правил, которые действуют для всех интеграций без исключения:

- **Единообразие клиентов.** Каждая интеграция реализуется как отдельный клиент в `app/integrations/<имя>` (или в `app/ai/` для AI-провайдера). Клиент инкапсулирует всю работу с внешним API: авторизацию, формирование запросов, обработку ошибок и ретраи. Бизнес-логика (сервисы) обращается к внешнему миру только через эти клиенты.
- **Токены только из настроек.** Ни один токен, ключ или секрет не хранится в коде. Все секреты читаются через Pydantic Settings из переменных окружения (файл `.env`, шаблон — в `.env.example`). В коде используется `settings.<имя>`, а не строковые литералы.
- **На Этапе 0 — заглушки.** В каркасе все клиенты являются заглушками (stub): они имеют согласованный интерфейс, но не выполняют реальных сетевых вызовов. Реальная реализация добавляется на соответствующем этапе разработки. Это позволяет писать и тестировать бизнес-логику до подключения внешних сервисов.
- **Логирование.** Все обращения к внешним сервисам и их результаты (особенно публикации и согласования) логируются.
- **Запрет на ложные кейсы.** Нельзя публиковать внешние картинки (найденные в интернете) как собственные производственные кейсы. Это правило учитывается на уровне интеграций поиска изображений и автопостинга.

## 1. Яндекс Диск

> **Статус: ✅ реализован на Этапе 2 (только чтение).** Бот сканирует Диск и сохраняет метаданные файлов в `MediaAsset`. На Этапе 2 файлы **не скачиваются**, vision-анализ изображений **не выполняется** — это задачи следующих этапов.

- **Назначение.** Хранилище медиа проекта. Команда загружает фото и видео в папки по проектам; бот сканирует Диск и читает имена файлов (метаданные). Скачивание медиа для анализа и использования в постах — задача более поздних этапов.
- **Что потребуется.** OAuth-токен приложения Яндекс с доступом к Диску. Переменные окружения:
  - `YANDEX_DISK_TOKEN` — OAuth-токен (`str | None`). Без токена обычный API работает (например, `GET /media-assets` отдаёт уже сохранённые активы из БД), а sync-эндпоинты возвращают `503`.
  - `YANDEX_DISK_BASE_URL` — базовый URL REST API, по умолчанию `https://cloud-api.yandex.net/v1/disk`.
  - `YANDEX_DISK_ROOT_PATH` — корневая папка бота на Диске, по умолчанию `/SMM_BOT`.
- **Этап подключения.** Этап 2 (Яндекс Диск).
- **Расположение клиента.** `app/integrations/yandex_disk/client.py`.
- **Протокол.** REST API Яндекс Диска (`cloud-api.yandex.net/v1/disk`) поверх `httpx`, авторизация OAuth-токеном. Реальные сетевые запросы выполняются **только при синхронизации** (sync); чтение списка активов из БД к Диску не обращается.
- **Структура папок на Диске.** Внутри `/SMM_BOT` для каждого проекта создаётся своя папка, а в ней — фиксированный набор подпапок:
  - `/SMM_BOT/01_TEEON` — проект TEEON (slug `teeon`);
  - `/SMM_BOT/02_Фабрика_сувениров` — проект «Фабрика сувениров» (slug `fabric-souvenirs`);
  - подпапки в каждом проекте: `01_Входящие_на_разбор`, `02_Одобренные_фото`, `03_Видео`, `04_Внешние_картинки_из_интернета`, `05_Использовано_в_постах`, `06_Нужно_переснять`.
  - На Этапе 2 сканируются **первые четыре** подпапки (`01`–`04`). Сопоставление проектов и путей живёт в `app/services/project_media_paths.py` (`get_project_disk_root`, `get_default_scan_folders`; неизвестный slug → `UnknownProjectError`).
- **Что делает клиент.** Методы: `list_resources` (`GET /resources`, парсит `_embedded.items`), `list_files_recursive` (рекурсивный обход, возвращает только файлы, `max_depth=3`), `get_download_url` (`GET /resources/download` → ссылка `href`). Каждый ресурс описывается dataclass `YandexDiskResource` (имя, путь, тип, MIME, размер, дата, `public_url`, свойства `is_file`/`is_dir`). Ошибки: `YandexDiskError` (базовая), `YandexDiskAuthError` (пустой токен либо HTTP `401`/`403`), `YandexDiskNotFoundError` (HTTP `404`). Получение самих ссылок на скачивание поддержано, но сами файлы на Этапе 2 не скачиваются.
- **Сервис синхронизации.** `app/services/yandex_disk_media_sync_service.py` (`YandexDiskMediaSyncService`): находит проект, перебирает папки сканирования, по пути классифицирует файлы (`04_Внешние_картинки_из_интернета` → `source_type=external_stock`, `license_type=external_needs_review`, `status=needs_license_review`; `02_Одобренные_фото` → `status=approved`; `03_Видео` → `status=approved_video`; остальное → `internal`/`company_owned`/`new`), извлекает теги через `MediaTaggingService.extract_tags_from_file_name` и делает upsert по `yandex_disk_path` (бизнес-уникальность — повторная синхронизация не плодит дубликаты). Отсутствующая папка попадает в `errors`, но синхронизация продолжается; `YandexDiskAuthError` прерывает весь процесс.
- **API.** Префикс `/media-assets` (`app/api/media_assets.py`): `GET /media-assets` (список с фильтрами `project_id`/`status`/`limit`/`offset`, работает без токена), `GET /media-assets/{id}` (`404`, если нет), `POST /media-assets/sync/project/{project_id}` и `POST /media-assets/sync/slug/{slug}` → `MediaAssetSyncResult` (`404` — нет проекта, `503` — нет/отклонён токен, `400` — неизвестная папка проекта). Ручной запуск: скрипт `app/scripts/sync_media.py` / команда `make sync-media project_slug=teeon`.
- **Хранение результата.** Метаданные сохраняются в `MediaAsset` (`file_name`, `yandex_disk_path` — unique + index, `source_type`, `license_type`, `status`, `tags`). Индексы добавлены миграцией `0002_media_asset_disk_path_indexes`.
- **Тесты.** Покрытие — без обращения к реальному API: сетевой слой подменяется `httpx.MockTransport`, БД поднимается на SQLite в памяти. Проверяются разбор `_embedded.items`, рекурсивный обход (только файлы), маппинг ошибок (`404`/пустой токен), сопоставление путей, идемпотентность upsert, классификация и тегирование, а также эндпоинты `/media-assets` (включая `503` без токена и `404` без проекта).

### Яндекс Диск: публичная ссылка SMM (альтернатива OAuth-токену)

> **Назначение.** Альтернативный режим: вместо приватного OAuth-токена бот читает медиа из **публичной папки** Яндекс Диска по public URL. **OAuth-токен в этом режиме не требуется.** Приватный режим остаётся и не меняется.

- **Переменные окружения:**
  - `YANDEX_DISK_PUBLIC_MODE` — `true`/`false` (по умолчанию `false`). Если `true`, `make sync-media` использует публичную ссылку.
  - `YANDEX_DISK_PUBLIC_SMM_URL` — публичная ссылка на папку SMM (например, `https://disk.yandex.ru/d/PYnchGnSLKW3yw`).
  - `YANDEX_DISK_PUBLIC_ROOT_FOLDER` — имя корневой папки внутри публичного ресурса (по умолчанию `SMM`).
- **Клиент.** `YandexDiskPublicClient` в `app/integrations/yandex_disk/client.py`: использует public-эндпоинты `GET /public/resources` и `GET /public/resources/download` **без OAuth-токена**. Методы `list_public_resources`, `list_public_files_recursive`, `get_public_download_url`; ресурс — dataclass `YandexDiskPublicResource` (имя, путь, тип, mime, размер, дата, `public_key`, `preview`, `file`, `media_type`, `embedded`).
- **Структура публичной папки.** Внутри `SMM` лежат проектные папки «Тион» и «Фабрика сувениров». Файлы именуются тегами (например, `futbolki-logotip-shelkografiya.jpg`, `hoodie-dtf-merch.mp4`).
- **Правила доступа (критично).** `app/services/project_media_paths.py` (`is_public_folder_allowed_for_project`, `get_public_project_folder_names`, `get_public_scan_roots`):
  - **teeon** — только папка «Тион»; **нельзя** брать из «Фабрика сувениров».
  - **fabric-souvenirs** — «Фабрика сувениров» **и** «Тион».
  - Сравнение имён нормализуется (нижний регистр, ё→е, без пробелов/дефисов), поэтому «Тион»/«TEEON»/«Tion» считаются одной папкой.
- **Сервис.** `app/services/public_yandex_disk_media_sync_service.py` (`PublicYandexDiskMediaSyncService`): находит проект, перебирает только разрешённые папки внутри SMM, рекурсивно собирает фото/видео (не фото/видео — пропускает), делает upsert `MediaAsset` по `yandex_disk_path` вида `public://yandex/<slug>/SMM/Тион/<файл>` (путь включает slug проекта — общий файл из «Тион» становится **отдельным** активом для каждого проекта, без коллизий) с `source_type=internal`, `license_type=company_owned`, `status=new`, тегами из имени файла. **Файлы не скачиваются** — только метаданные. Правило доступа проверяется по **полному пути** файла: чужая проектная папка, вложенная в разрешённую (например, «Фабрика сувениров» внутри «Тион»), для teeon отсекается (`is_public_path_allowed_for_project`).
- **API.** `POST /media-assets/sync/public/project/{project_id}` и `POST /media-assets/sync/public/slug/{slug}` → `MediaAssetSyncResult` (`404` — нет проекта, `503` — не задана публичная ссылка). Ручной запуск: `make sync-public-media project_slug=teeon` (или `python -m app.scripts.sync_media --project-slug teeon --public`).
- **Тесты.** Без сети: публичный клиент через `httpx.MockTransport`, сервис через fake-клиент. Проверяются разбор public `_embedded.items`, правила доступа (teeon ⊄ Фабрика), пропуск не-медиа, идемпотентность, и API с подменённым сервисом (включая `503` без публичной ссылки).
- **Подробности.** См. `Докс/18_Публичная_папка_Яндекс_Диска.md`.

## 2. Telegram

- **Назначение.** Двойная роль: (1) интерфейс согласования — бот присылает ответственному сгенерированные посты с кнопками «Опубликовать / Доработать / Отклонить»; (2) канал автопостинга — публикация одобренных постов в Telegram-каналы проектов.
- **Что потребуется.** Токен бота от @BotFather. Переменная окружения: `TELEGRAM_BOT_TOKEN`. Дополнительно — ID каналов проектов и ID чата(ов) согласующих.
- **Этап подключения.** Этап 6 (интерфейс согласования), затем расширяется на Этапе 7 (автопостинг).
- **Расположение клиента.** `app/integrations/telegram`.
- **Что делает клиент.** Отправка сообщений с inline-кнопками, обработка callback-ответов согласующего, отправка медиа и текста (`Post.telegram_text`, `Post.hashtags`) в целевой канал, фиксация результата публикации (`published_at`, `status`).

## 3. VK

- **Назначение.** Автопостинг одобренных публикаций в сообщества ВКонтакте проектов.
- **Что потребуется.** Access-токен сообщества (с правами на публикацию записей и загрузку фото). Переменная окружения: `VK_ACCESS_TOKEN`. Дополнительно — ID групп проектов и версия API VK.
- **Этап подключения.** Этап 7 (автопостинг).
- **Расположение клиента.** `app/integrations/vk`.
- **Что делает клиент.** Загрузка медиа на сервер VK, создание записи на стене сообщества с текстом `Post.vk_text` и вложениями, возврат идентификатора публикации для последующего сбора аналитики.

## 4. Instagram

- **Назначение.** Автопостинг публикаций в Instagram проектов. Подключается позже остальных каналов из-за более сложных требований платформы.
- **Что потребуется.** Access-токен Instagram Graph API (через бизнес-аккаунт, связанный с Facebook Page). Переменная окружения: `INSTAGRAM_ACCESS_TOKEN`. Дополнительно — ID бизнес-аккаунтов и требование к публично доступным URL медиа.
- **Этап подключения.** Этап 7 (автопостинг, реализуется позже Telegram и VK).
- **Расположение клиента.** `app/integrations/instagram`.
- **Что делает клиент.** Создание медиа-контейнера, публикация контейнера, использование текста `Post.instagram_text` и хэштегов, обработка ограничений платформы (форматы, лимиты на хэштеги, требования к изображениям).

## 5. AI-провайдер

- **Назначение.** Ядро интеллектуальной части бота: генерация идей и тем публикаций, написание текстов под каждую соцсеть, анализ изображений из хранилища (распознавание изделий, технологий, деталей для тегирования).
- **Что потребуется.** Имя провайдера и ключ доступа к его API. Переменные окружения: `AI_PROVIDER` (выбор провайдера) и `AI_API_KEY` (ключ доступа). Опционально — модель и параметры генерации.
- **Этап подключения.** Используется на нескольких этапах: Этап 3 (анализ медиа), Этап 4 (выбор тем), Этап 5 (генерация постов). Полноценная реализация начинается с Этапа 3.
- **Расположение клиента.** `app/ai/client.py`.
- **Что делает клиент.** Унифицированный интерфейс к LLM/мультимодальной модели: текстовая генерация (темы, тексты постов, хэштеги) и анализ изображений (теги для `MediaAsset.tags`). Сервисы `TopicSelectionService`, `MediaTaggingService` и `PostGenerationService` обращаются к AI только через этот клиент.

## 6. SEO / тренды

- **Назначение.** Источник данных для обоснованного выбора тем: ключевые слова, их частотность, сезонность спроса, тренды. Помогает боту приоритизировать темы в рамках заданных человеком направлений.
- **Что потребуется.** Доступ к источнику(ам) данных по ключевым словам и трендам (например, сервисы статистики поисковых запросов). Переменная окружения для ключа — `SEO_API_KEY` (точный набор уточняется при подключении, возможно несколько источников).
- **Этап подключения.** Этап 4 (выбор тем).
- **Расположение клиента.** `app/integrations/seo`.
- **Что делает клиент.** Получение частотности и сезонности по списку запросов, оценка трендов. Данные используются для заполнения `Topic.seo_keywords` и расчёта `Topic.priority_score` совместно с `business_priority` и наличием подходящего медиа.

## 7. Аналитика

- **Назначение.** Сбор статистики опубликованных постов из соцсетей (охваты, просмотры, лайки, комментарии, репосты) для оценки эффективности и обратной связи в выбор будущих тем.
- **Что потребуется.** Доступ к API статистики используемых платформ. Отдельный токен не вводится — переиспользуются токены соцсетей (`TELEGRAM_BOT_TOKEN`, `VK_ACCESS_TOKEN`, `INSTAGRAM_ACCESS_TOKEN`).
- **Этап подключения.** Этап 8 (аналитика).
- **Расположение клиента.** `app/integrations/analytics`.
- **Что делает клиент.** Запрос метрик по идентификаторам опубликованных записей, нормализация показателей разных платформ к единому виду, сохранение статистики для анализа. Эти данные служат сигналом для `TopicSelectionService` при выборе следующих тем.
