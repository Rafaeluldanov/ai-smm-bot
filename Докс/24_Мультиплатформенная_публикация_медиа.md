# 24. Мультиплатформенная публикация медиа (Yandex Disk → platform adapter)

Единая архитектура «медиа с Яндекс Диска → capability-слой → platform adapter →
publish/preview» для VK, Telegram, Instagram, YouTube, RuTube и будущих платформ.
Медиа берётся с Яндекс Диска по тегам, группируется, обрастает SEO-текстом со
ссылкой на сайт и раскладывается по платформам согласно их возможностям.

## Платформы и статус

| Платформа | Текст | Фото | Альбом фото | Видео | Live-статус |
|-----------|:----:|:----:|:-----------:|:-----:|-------------|
| **VK**       | ✅ | ✅ | ✅ | ⏳ | **live-ready** (текст + фото-группа) |
| **Telegram** | ✅ | ✅ | ✅ | ⏳ | **live-ready** (текст + фото-альбом) |
| **Instagram**| ✅ | ✅ | ✅ (carousel) | ✅ (reels) | skeleton / dry-run |
| **YouTube**  | описание | — | — | ✅ (shorts/video) | skeleton / dry-run |
| **RuTube**   | описание | — | — | ✅ | skeleton / dry-run |

⏳ — видео пока пропускается с предупреждением (upload не реализован).
**skeleton / dry-run** — preview показывает, что ушло бы, но live-публикация
возвращает понятную ошибку `Live publishing for <platform> is not implemented yet`.

## Матрица возможностей (capability-слой)

`backend/app/integrations/platform_capabilities.py` — единственный источник правды
о возможностях платформ. Каждая платформа описана `PlatformCapabilities`:
`supports_text/image/image_group/video/video_group`, `max_images`, `max_videos`,
`max_text_length`, `live_flag_name`, `notes`. Клиенты платформ **не содержат**
capability-логики — только транспорт.

`route_media(caps, media_items)` решает, что уйдёт на платформу:
- поддерживает альбом фото и фото ≥ 2 → `image_group` (до `max_images`, лишнее —
  предупреждение);
- поддерживает фото, но не альбом → одно фото;
- платформа видео-only, а есть только фото → `unsupported_media_reason`;
- платформа фото-only, а есть только видео → text-only fallback + предупреждение;
- mixed: фото-платформы берут фото и пропускают видео; видео-платформы берут видео.

## Яндекс Диск → байты

Общий модуль `backend/app/integrations/media_attachments.py`:
`load_item_bytes(item, downloader)` (локальная enhanced-копия или
`public://yandex/...` через `MediaDownloadService.download_public_media`),
`maybe_convert_heic(...)` (HEIC/HEIF → JPEG **в памяти**, оригинал не
перезаписывается), `is_image`/`is_video`/`content_type`/`sanitize_filename`.
Используется VK- и Telegram-клиентами; новые платформы подключаются так же.

## Токены (в .env, НЕ в docs)

- VK: `VK_ACCESS_TOKEN`, `VK_DEFAULT_GROUP_ID`
- Telegram: `TELEGRAM_BOT_TOKEN`, `TELEGRAM_DEFAULT_CHANNEL_ID`
- Instagram: `INSTAGRAM_ACCESS_TOKEN`, `INSTAGRAM_BUSINESS_ACCOUNT_ID`
- YouTube: `YOUTUBE_ACCESS_TOKEN`, `YOUTUBE_CHANNEL_ID`
- RuTube: `RUTUBE_ACCESS_TOKEN`, `RUTUBE_CHANNEL_ID`
- Общий источник медиа: `YANDEX_DISK_PUBLIC_SMM_URL`

Токены НИКОГДА не логируются, не попадают в `raw` ответа и в тексты ошибок.

## Безопасность

- Все live-флаги по умолчанию **false**
  (`*_LIVE_PUBLISHING_ENABLED`): без флага отправка невозможна.
- Instagram/YouTube/RuTube даже с флагом live пока **не реализованы** — понятная
  ошибка; dry-run/preview работает.
- **Dry-run first:** сначала `--dry-run`, только потом ручная публикация.
- `publish-due` во время разработки/тестов **не использовать**.
- В тестах реальной сети нет (`httpx.MockTransport`/fake).

## Команды

```bash
# Превью медиа по всем платформам (ничего не создаёт и не шлёт)
make media-platform-preview project_slug=teeon tag=футболка \
  platforms="telegram,vk,instagram,youtube,rutube"

# Создать пост из группы медиа
make media-group-post project_slug=teeon tag=футболка
export POST_ID=<new_id>

# Dry-run preview поста по платформам
make publish-preview post_id=$POST_ID
# = python -m app.scripts.publish_post --post-id $POST_ID --dry-run

# Ручная публикация ОДНОГО поста в live-ready платформу (только вручную)
TELEGRAM_LIVE_PUBLISHING_ENABLED=true VK_LIVE_PUBLISHING_ENABLED=false \
  python -m app.scripts.publish_post --post-id $POST_ID --platform telegram
```

## API

- `GET /post-publications/platform-capabilities` — список платформ и их
  возможностей (текст/фото/видео, лимиты, live-флаги, live_implemented).
- `POST /post-publications/preview/{post_id}` — dry-run preview по платформам:
  `platform`, `live_enabled`, `target_id`, `media_kind`, `media_count`,
  `media_asset_ids`, `would_attach_media`, `media_warnings`,
  `unsupported_media_reason`, `platform_capabilities`. Ничего не отправляет.

## Как добавить платформу

1. Запись в `build_platform_capabilities` (возможности + лимиты + live-флаг).
2. Клиент `integrations/<platform>/client.py`, реализующий `PublishingClient`
   (`platform`, `live_enabled`, `publish_post`; `live_implemented=False` для
   скелета). live-флаг — в `config.py` + `.env.example` (default false).
3. Регистрация клиента в `deps.get_publication_platform_registry` +
   `default_targets`.
4. Тесты: live off без сети, отсутствие токена, live-not-implemented, token safety.

CRM: тип ресурса (`resource_type`) — обычная строка (`instagram`/`youtube`/`rutube`
уже в списке), миграция не нужна.
