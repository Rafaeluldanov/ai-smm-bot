# Проверки и тестирование

Этот документ описывает систему контроля качества проекта «AI-SMM-бот». Качество кода обеспечивают три инструмента — **ruff** (линтинг и форматирование), **mypy** (статическая типизация) и **pytest** (тесты). Все они объединены командой `make check`, которая является обязательными «воротами качества» перед переходом на следующий этап разработки.

Все команды запускаются из каталога `backend/` (или через корневой `Makefile`, который сам переходит в нужную директорию). Настройки инструментов хранятся в `backend/pyproject.toml`.

---

## 1. ruff — линтинг и форматирование

`ruff` — быстрый линтер и форматтер для Python, заменяющий связку flake8 + isort + black. Используется как для проверки стиля, так и для автоматического форматирования.

**Какие правила включены** (секция `[tool.ruff.lint]` в `pyproject.toml`):

| Набор | Что проверяет |
|-------|---------------|
| `E`, `W` | Ошибки и предупреждения pycodestyle (отступы, пробелы, длина строки) |
| `F` | pyflakes — неиспользуемые импорты и переменные, неопределённые имена |
| `I` | isort — порядок и группировка импортов |
| `UP` | pyupgrade — устаревшие синтаксические конструкции |
| `B` | flake8-bugbear — потенциальные баги и сомнительные паттерны |
| `C4` | оптимизация comprehension-выражений |
| `SIM` | flake8-simplify — упрощение логики |

Длина строки — 100 символов. Целевая версия — Python 3.11.

**Команды:**

```bash
make lint      # ruff check . — проверка без изменений (только отчёт)
make format    # ruff format . + ruff check --fix . — автоформатирование и автоисправление
```

`make lint` ничего не меняет в файлах и завершается с ненулевым кодом при наличии нарушений. `make format` приводит код к единому стилю и исправляет всё, что можно исправить автоматически (например, сортировку импортов).

## 2. mypy — статическая типизация

`mypy` проверяет аннотации типов до запуска кода и ловит ошибки, которые иначе проявились бы только в рантайме (неверные аргументы, `None` там, где ожидается объект, несовместимые типы).

Проект использует **строгий режим** (`strict = true` в секции `[tool.mypy]`), что включает в том числе:

- `disallow_untyped_defs` — все функции и методы должны иметь аннотации;
- `disallow_any_generics` — запрет «голых» дженериков (например, `list` вместо `list[str]`);
- `warn_return_any`, `warn_unused_ignores` — предупреждения о потере типов и лишних `# type: ignore`;
- `check_untyped_defs` — проверка тел функций без аннотаций.

Для сторонних библиотек без типовых стабов (например, отдельные клиенты интеграций на этапах-заглушках) допускается точечное послабление через `[[tool.mypy.overrides]]` с `ignore_missing_imports = true`. Это не должно затрагивать собственный код пакета `app`.

**Команда:**

```bash
make typecheck   # mypy app
```

## 3. pytest — модульные и интеграционные тесты

Тесты лежат в каталоге **`backend/tests`**. Структура повторяет структуру пакета `app`:

- модульные тесты сервисов и утилит (`tests/services`, `tests/utils`);
- тесты схем и моделей (`tests/schemas`, `tests/models`);
- интеграционные тесты API через `TestClient` FastAPI (`tests/api`);
- общие фикстуры — в `tests/conftest.py`.

Настройки в секции `[tool.pytest.ini_options]`: `testpaths = ["tests"]`, корень для импортов — `app`. Интеграционные тесты, требующие PostgreSQL/Redis, помечаются маркером `@pytest.mark.integration` и при необходимости поднимаются через `docker-compose`.

**Команда:**

```bash
make test    # pytest -q
```

### Тесты Этапа 0

На каркасе уже работают четыре базовых проверки:

1. **`GET /health`** — эндпоинт возвращает HTTP 200 и тело `{"status": "ok", "service": "ai-smm-bot"}`. Проверяется через `TestClient`.
2. **Загрузка конфигурации** — `Settings` (Pydantic Settings) корректно читается из окружения/`.env`, обязательные поля заданы, значения по умолчанию применяются.
3. **`MediaTaggingService`** — разбор имени файла. Контрольный пример:
   `«Худи с карманом с шелкографией и жаккардами.jpg»` →
   - изделие: `худи`;
   - технологии: `шелкография`;
   - детали: `карман`, `жаккард`;
   - предполагаемый проект: `TEEON`;
   - теги/темы: «худи с логотипом», «корпоративный мерч», «шелкография на худи».
4. **Утилиты текста** — функции из `app/utils` (нормализация регистра, очистка от лишних пробелов, слугификация, формирование хэштегов) дают ожидаемый результат на наборе входных строк.

## 4. make check — «ворота качества»

`make check` — главная команда контроля. Она последовательно запускает все три проверки:

```bash
make check    # ruff check + mypy app + pytest
```

Порядок: сначала `ruff check` (быстро отсекает стилевые проблемы), затем `mypy` (типы), затем `pytest` (поведение). Команда завершается ошибкой, если падает хотя бы один шаг.

**Правило проекта:** перед переходом к следующему этапу разработки `make check` обязан проходить полностью (зелёный статус). Это гарантирует, что каждый этап остаётся стабильной точкой, на которую можно опираться дальше. Сложные действия дополнительно логируются, но логи не заменяют тесты.

## 5. Что проверяется на каждом этапе

В таблице ниже показано, какие проверки и тесты добавляются по мере развития проекта. На каждом этапе `make check` включает всё накопленное ранее.

| Этап | Что добавляется в коде | Какие проверки и тесты добавляются |
|------|------------------------|------------------------------------|
| **0. Каркас** | FastAPI, конфиг, модели, схемы, сервисы-заглушки, клиенты-заглушки | `ruff`, `mypy --strict`, `pytest`; тесты: `/health`, загрузка конфига, `MediaTaggingService` (разбор имени файла), утилиты текста |
| **1. Проекты** | CRUD `Project`, миграции Alembic | Тесты CRUD-эндпоинтов проектов; проверка `slug` (уникальность); тест миграций (upgrade/downgrade) |
| **2. Яндекс Диск** | Клиент `app/integrations/yandex_disk`, сканирование папок | Юнит-тесты клиента на замоканных ответах API; тест сопоставления файлов с проектами; токен только из `.env` |
| **3. Анализ медиа** | Расширение `MediaTaggingService`, анализ изображений | Тесты тегирования по именам и метаданным; проверка заполнения `tags`, `title`, `description`, `status` у `MediaAsset` |
| **4. Выбор тем** | `TopicSelectionService` (рабочая логика), кластеризация, скоринг | Тесты расчёта `priority_score`, учёта `business_priority`, `seo_keywords`, сезонности; тест дедупликации тем |
| **5. Генерация постов** | `PostGenerationService`, AI-клиент | Тесты генерации `telegram_text`/`vk_text`/`instagram_text`, формирования `hashtags`; проверка, что выбран медиаассет под тему |
| **6. Согласование** | Статусы `Post`, отправка на ревью | Тесты переходов статусов (draft → on_review → approved/rejected); проверка нотификаций (мок) |
| **7. Автопостинг** | Клиенты Telegram/VK/Instagram, scheduler | Тесты планировщика (`scheduled_at` → `published_at`); тесты публикации на моках; запрет публикации без `approved` |
| **8. Аналитика** | Сбор статистики публикаций | Тесты агрегации метрик; влияние статистики на скоринг тем; тест отчётов |
| **9. Поиск внешних картинок** | Внешние источники медиа | Тесты `source_type`/`license_type`; **проверка запрета** публиковать внешние картинки как собственные кейсы |
| **10. Автономный режим** | Полный цикл без ручного запуска | End-to-end тест цепочки тема → пост → согласование → публикация; тесты идемпотентности и логирования сложных действий |

### Кратко

- **Линт/формат:** `make lint`, `make format`.
- **Типы:** `make typecheck` (строгий mypy).
- **Тесты:** `make test` (каталог `backend/tests`).
- **Перед следующим этапом:** `make check` должен быть зелёным.
