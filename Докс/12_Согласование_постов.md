# Согласование постов (Этап 6)

Документ описывает backend-логику **согласования черновиков** постов. На этом этапе реальный Telegram Bot API **не** подключается: готовится только интерфейсный слой (preview-карточка с кнопками), а решения принимаются через API/CLI. Публикация, автопостинг и AI не используются.

## Жизненный цикл черновика

Пост создаётся на Этапе 5 со статусом `draft` (есть медиа) или `needs_media` (медиа нет). Дальше им управляет согласующий через `PostReviewService`:

```
needs_media ──(добавить медиа / return_to_draft)──► draft
   draft ──submit──► needs_review ──approve──► approved ──► (scheduled, Этап 7)
     ▲                  │   │                     │
     │             reject│   │request_changes      │reject / request_changes
     │                  ▼   ▼                     ▼
  return_to_draft ◄─ rejected                   draft
```

Каждое действие:

1. проверяет переход статуса через `post_status_service.validate_transition` (безопасные переходы);
2. меняет `Post.status`;
3. пишет запись в журнал `PostReviewAction` (`from_status`, `to_status`, `comment`, `actor_name`, `actor_role`, `payload`).

## Решения: approve / reject / request changes / return to draft

`PostReviewService` (`backend/app/services/post_review_service.py`):

| Метод | Действие | Переход статуса |
|-------|----------|-----------------|
| `submit_for_review` | отправить на согласование | `draft`/`needs_media` → `needs_review` (нужно медиа) |
| `approve_post` | одобрить | `needs_review`/`draft` → `approved` |
| `reject_post` | отклонить | `draft`/`needs_review`/`approved`/`needs_media` → `rejected` |
| `request_changes` | вернуть на доработку | `needs_review`/`approved` → `draft` |
| `return_to_draft` | вернуть в черновик | `rejected`/`approved`/`needs_review`/`needs_media` → `draft` |
| `edit_post_texts` | ручная правка текстов/медиа | без смены (или `rejected` → `draft`) |
| `add_comment` | комментарий | без смены статуса |

Бизнес-правила:

- **нельзя отправить на согласование без медиа** (`submit_for_review` при пустом `media_asset_id` → 409);
- **нельзя редактировать `published`** (разрешено только `draft`/`needs_review`/`needs_media`/`rejected`);
- **правка отклонённого поста переводит его в `draft`** (документируется в `payload`);
- запрещённый переход → ошибка `InvalidPostStatusTransitionError` (API → 409); неизвестный статус → 422; нет поста → 404.

## Журнал действий

Журнал — таблица `post_review_actions` (модель `PostReviewAction`). Это audit-log: записи не меняются и не удаляются. По нему строится **таймлайн** (`get_timeline`) — список действий в хронологическом порядке и текущий статус поста.

`payload` хранит детали: для правок — `changed_fields`, `before`, `after` (версии текста/полей до и после); для решений значимы `from_status`/`to_status` и комментарий. Подробности модели — в [`Докс/06_Модель_данных.md`](./06_Модель_данных.md) (раздел «PostReviewAction»).

`build_review_card` собирает карточку поста для согласования: тексты, хэштеги, SEO, число действий, время последнего действия и предупреждения (нет медиа, статус `needs_media`, заполнены не все площадки).

## Telegram preview

`TelegramReviewInterface` (`backend/app/integrations/telegram/review_interface.py`) строит структуру будущего сообщения **без отправки**:

- `build_review_message(card)` → `TelegramReviewMessage` (заголовок, статус, превью текста, хэштеги, предупреждения + кнопки);
- `build_buttons(post_id, status)` → кнопки под текущий статус: `approve`, `reject`, `request_changes`, `return_to_draft` (зависят от статуса) плюс всегда `open_post` и `add_comment`.

Например, для `needs_review` доступны approve/reject/request_changes/return_to_draft, для `rejected` — только return_to_draft. `callback_data` имеет вид `postreview:<action>:<post_id>` — на Этапе 7 это будет обрабатываться реальным ботом.

## API и команды

REST API `/post-reviews` (`backend/app/api/post_reviews.py`):

- `GET /post-reviews/{post_id}/card` — карточка поста;
- `GET /post-reviews/{post_id}/timeline` — история действий;
- `GET /post-reviews/{post_id}/telegram-preview` — превью карточки (без отправки);
- `POST /post-reviews/{post_id}/submit|approve|reject|request-changes|return-to-draft` — решения;
- `PATCH /post-reviews/{post_id}/edit` — ручная правка;
- `POST /post-reviews/{post_id}/comment` — комментарий.

Коды: 404 — поста нет; 422 — неизвестный статус; 409 — запрещённый переход/действие.

CLI:

```bash
make review-post post_id=1 action=submit
make approve-post post_id=1
make reject-post post_id=1
```

## Чего НЕ делается на Этапе 6

- **нет реальной отправки в Telegram** — только структура карточки и кнопок;
- **нет публикации и автопостинга** (это Этап 7);
- **нет AI и внешних сетевых запросов**;
- соцсети (Telegram/VK/Instagram) не подключаются.
