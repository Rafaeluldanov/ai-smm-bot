# Автономный режим (Этап 10)

Документ описывает **автономный pipeline** — управляемый прогон, связывающий все реализованные модули в единый цикл подготовки работы SMM-менеджера. Реальные публикации и AI выполняются только при явном разрешении; сеть в тестах не вызывается, fake-провайдеры заменяют реальные стоки/соцсети.

## Режимы

| Режим | Что делает | Что запрещено |
|-------|-----------|----------------|
| `dry_run` | Создаёт прогон и шаги, описывает, что было бы сделано | Не создаёт темы/посты/публикации |
| `semi_auto` | Выбирает темы, генерирует посты, отправляет черновики на согласование (`needs_review`) | Авто-одобрение/планирование/публикация |
| `auto_generate` | Генерация постов (+ ревью) | Планирование, публикация |
| `auto_schedule` | Планирует **только approved** посты | Публикация без `allow_auto_publish` |
| `auto_publish` | Публикует (Telegram/VK) | Только при `allow_auto_publish=true` |

Эффективные настройки = **возможности режима ∩ явные настройки** (`AutonomousModeSettings`): `allow_external_images`, `allow_auto_approve`, `allow_auto_schedule`, `allow_auto_publish`, `require_human_review`, `platforms`, `dry_run`. Режим задаёт потолок, настройки могут только включить разрешённое режимом.

## Полный pipeline

1. **resolve project** — по `project_id`/`project_slug` (нет → 404).
2. **validate** — `AutonomousSafetyService.validate_request` (неверный режим/настройки → 422).
3. **create run** — `AutonomousRun` (status `running`).
4. **select_topics** — `TopicSelectionService.select_topics_for_project`.
5. **build_content_plan** — `TopicSelectionService.build_weekly_content_plan`.
6. **generate_posts** — `PostGenerationService.generate_weekly_posts` (draft / needs_media).
7. **select_media** — информационный шаг (сколько постов с медиа / needs_media).
8. **search_external_images** — для `needs_media` постов при `allow_external_images` (`ExternalImageSearchService.search_for_post`).
9. **submit_for_review** — черновики уходят на согласование (или авто-одобрение при `allow_auto_approve` и разрешении safety).
10. **schedule_posts** — при `allow_auto_schedule` планирует approved-посты.
11. **publish_posts** — при `allow_auto_publish` публикует approved/scheduled.
12. **collect_analytics** — на прогоне реальная аналитика не запрашивается (Этап 8).
13. **build_report** — сводка, предупреждения, `next_actions`.

Каждый шаг пишет `AutonomousRunStep` (статус `completed`/`skipped`/`failed`, вход/выход, предупреждения). **Одна проблема не роняет весь прогон** — он завершается со статусом `completed_with_warnings`. Критическая ошибка (нет проекта, неверный режим) возникает до создания прогона.

## Guardrails

`AutonomousSafetyService`:

- **require_human_review** (по умолчанию true) — нельзя авто-одобрять посты; в `dry_run`/`semi_auto` всегда true.
- **needs_media** — нельзя авто-одобрять/планировать/публиковать.
- **внешний сток** — нельзя авто-одобрять при лицензии `external_needs_review` или статусе `needs_license_review`.
- **auto_schedule** — планирует только `approved` посты.
- **auto_publish** — публикует только `approved`/`scheduled` и только при `allow_auto_publish=true`.

## Что можно автоматизировать, а что требует человека

- **Можно автоматизировать:** выбор тем, контент-план, генерацию черновиков, подбор медиа, поиск внешних картинок, отправку на ревью, (по разрешению) планирование и публикацию.
- **Требует человека:** одобрение постов (по умолчанию), проверку внешних картинок (лицензии, «не наш кейс»), решение о реальной публикации (`allow_auto_publish`), досъёмку для `needs_media`.

Отчёт (`build_report`) формирует `next_actions` — конкретные шаги для менеджера: где согласовать, где добавить медиа, что проверить.

## API и команды

REST API `/autonomous-runs` (`backend/app/api/autonomous_runs.py`):

- `GET /autonomous-runs?project_id=&status=&mode=` — список прогонов;
- `GET /autonomous-runs/{run_id}` — прогон (404);
- `GET /autonomous-runs/{run_id}/steps` — шаги (404);
- `GET /autonomous-runs/{run_id}/report` — отчёт с `next_actions` (404);
- `POST /autonomous-runs/run` — запустить (404 — нет проекта; 422 — режим/настройки);
- `POST /autonomous-runs/dry-run` — сухой прогон;
- `POST /autonomous-runs/run/project/{project_id}` и `POST /autonomous-runs/run/slug/{slug}`.

CLI:

```bash
make autonomous-run project_slug=teeon
make autonomous-dry-run project_slug=teeon
make autonomous-report run_id=1
```

Планировщик: `autonomous_weekly_run_job(db, service, project_slug, request=None)` — еженедельный прогон одним проходом (по умолчанию `semi_auto`).

## Ограничения Этапа 10

- **в тестах нет реальных API и сети** — публикация через fake-клиентов;
- **`auto_publish` работает только при явном `allow_auto_publish`**;
- Instagram и AI не подключены;
- бесконечных фоновых циклов нет — прогон выполняется одним проходом.
